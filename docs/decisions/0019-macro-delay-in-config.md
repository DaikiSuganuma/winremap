# ADR 0019: macro_delay_ms を設定ファイルに昇格する

- ステータス: 承認
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

実機検証（ADR 0016 拡充後のデバッグログ）により、マクロの間欠失敗の原因が確定した: 注入イベント列は毎回完全・外部干渉なしで、**WinUI 版メモ帳が一括注入のストローク列を取りこぼす**（処理が追いつかない）ことが原因。`--macro-delay 8` のペーシング（ADR 0018)で安定することがオーナーにより確認された。恒常運用のため、毎回 CLI フラグを付けずに済む手段が必要（ADR 0018 で予告した再検討）。

## 決定

1. TOML のトップレベルに **`macro_delay_ms`（整数 0〜15、既定 0）** を追加する
2. 優先順位は **CLI `--macro-delay` > 設定ファイル > 既定 0**。CLI 指定は実験用オーバーライドとして、トレイからのリロード後も維持される
3. 値は `RemapTable` に載せて運搬し、リロード時にルールと一緒に適用される
4. `examples/suganuma.toml` に `macro_delay_ms = 8` を設定（実測で安定を確認した値）

## 理由

- 環境（使用アプリ）に依存する恒常設定であり、起動方法に依存する CLI フラグより設定ファイルが適切
- CLI を上書きとして残すことで、値の調整実験がリロード無しで行える

## 却下した代替案

- **既定値を 8 にする**: 大多数の従来型アプリでは不要であり、マクロ実行中の全システム入力遅延（3 ストロークで約 16ms）を全ユーザーに払わせる理由がない
- **アプリ別の delay 指定**: 表現力は上がるが、現時点のニーズは全体一律で足りる。必要になったら keymap 単位の設定として再検討
