# ADR 0016: --debug のキーイベントログ（リングバッファ経由）

- ステータス: 承認（オーナー指示）
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

`--debug` で「入力したキーと winremap が何をしようとしたか」をコンソールに出力したい。ただしフックコールバック内の I/O・ヒープ確保は禁止（AGENTS.md 不変条件 2）であり、コールバックから直接 `println!` はできない。また不変条件 6（キーロガー化の禁止）との整合も必要。

## 決定

1. **記録**: コールバックは固定長 128 要素の thread_local リングバッファに Copy な小さいレコード（入力コンボ、プレフィックス、決定内容）を書くだけ。あふれた分は件数のみカウント
2. **起床**: キーストロークは自スレッドにメッセージを発生させないため、そのままではメッセージループが眠り続けログが遅延する。debug 有効時のみ `PostThreadMessageW(WM_APP)` で自スレッドを起こす（軽量・非ブロッキング）
3. **出力**: メッセージループ側（各メッセージ処理後のコールバック）でバッファを排出し、i18n 経由で整形して `println!` する
4. **記録対象は keydown の決定のみ**（素通し / 置換 / 単キー差し替え / マクロ実行 / プレフィックス待機 / シーケンス握りつぶし）。keyup・キーリピート・修飾キー単体は記録しない（ノイズ削減）
5. 不変条件 6 との整合: 出力は**キー名レベル**のみ、既定 OFF（`--debug` 指定時のみ）、**永続化しない**（コンソール出力のみでファイル保存機能は作らない）

## 理由

- 固定長リング + Copy レコードで、コールバック内はメモリ書き込みのみ（確保・ロック・I/O ゼロ）を維持できる
- フックとメッセージループは同一スレッドなので、リングは thread_local で足り、同期プリミティブが不要

## 却下した代替案

- **コールバックから直接 println!**: 不変条件 2 違反。コンソールが詰まるとフックがタイムアウトし、Keyhac の問題を再現してしまう
- **チャネル / Mutex 付きキュー**: ロック待ちの可能性がコールバックに入る。同一スレッドなので不要
- **専用ログスレッド**: スレッド間共有が必要になり複雑化。debug 用途には過剰
