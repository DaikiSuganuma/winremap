# ADR 0015: Alt/Win のメニュー誤発動をマスクキーで防止する

- ステータス: 承認（ADR 0005 の既知の制限を解消）
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

`"A-a" = ["C-Home", "C-S-End"]`（全選択マクロ）が機能しない不具合が実機で確認された。原因は Windows の「Alt / Win の単独タップでメニューバー / スタートメニューを開く」挙動。winremap はチョードの元キー（この例では `a`）を抑止するため、システムから見ると **Alt down → Alt up だけが起きた**ことになり、メニューバーが活性化して後続の注入キーが編集領域に届かない。

同じ理由で次も影響を受けていた: Alt チョードの修飾一時解除（ADR 0005 の lift）、復元後の物理 Alt 解放、`A-x` プレフィックス押下後の物理 Alt 解放。

## 決定

Alt / Win の down と up の間に**マスクキー（予約 VK `0xFF` のタップ）**を注入し、単独タップ判定を成立させない。Keyhac / AutoHotkey（MenuMaskKey）と同じ手法。

1. **注入側**（`sender.rs`）: `send_exact_down` / `send_sequence` で Alt/Win を一時解除する場合、解除イベントの直前に同一バッチ内でマスクをタップする
2. **物理解放側**（`hook.rs`）: Alt/Win を含むチョードを抑止した時点で `MENU_GUARD`（thread_local のビットフラグ）を立てる。ガードが立った状態で物理 Alt/Win の keyup が来たら、その keyup を抑止し `[マスク down, マスク up, 修飾キー up]` を 1 バッチで注入し直す（順序保証のため）。注入し直す up には `MARKER_REMAP` を付け、修飾キー追跡が追随する

## 理由

- マスクの位置が重要（up より**前**）なため、物理 up をそのまま通して別途マスクを注入すると順序が保証されない。抑止して 1 バッチで置き換えるのが確実
- VK 0xFF は予約値でアプリには無視されるが、OS の単独タップ判定は破れる。実績のある定番手法
- ガードは「winremap がチョードを消費した場合」のみ立つので、ユーザーが Alt を普通に単独タップしてメニューを開く操作は今までどおり機能する

## 却下した代替案

- **既知の制限として放置**（ADR 0005 当時の判断）: fakeymacs 移植で Alt 系チョード（A-a / A-f / A-x ...）が主役になり、実害が確定したため撤回
- **Ctrl 等の実在キーでマスク**: アプリが Ctrl 押下に反応する副作用がある（AutoHotkey の旧既定で既知の問題）
- **メニュー活性化を検知して閉じる**: 事後対応では一瞬のフォーカス移動が見えるうえ、検知自体が不確実
