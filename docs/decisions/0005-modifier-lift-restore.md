# ADR 0005: 修飾キーの一時解除・復元シーケンス

- ステータス: 承認
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

`"C-h" = "Back"` のような修飾キー付きルールでは、物理 Ctrl が押下されたまま Backspace を送ると、アプリには Ctrl+Backspace（多くのアプリで単語削除）が届いてしまい、発端の問題（1文字削除にしたい）が解決しない。fakeymacs / Keyhac も同種の「修飾キーの一時解除」を行っている。

## 決定

`sender.rs` に以下のシーケンスを実装した。修飾キーは**左右個別のビットセット（SideMods）**で追跡し、解除は「実際に押下されている側のキー」を対象にする。

1. **押下時**（`send_exact_down`）: ターゲットに不要な押下中修飾キーの UP → ターゲットに必要だが未押下の修飾キーの DOWN（左側キーを使用）→ ターゲットキー DOWN を **1 回の `SendInput` 呼び出し**（固定長スタック配列）で送出
2. **リピート時**: ターゲットキー DOWN のみ再送出（修飾状態は初回押下時に調整済み）
3. **解放時**（`send_exact_up`）: ターゲットキー UP → 追加した修飾キーの UP → 解除した修飾キーのうち**現在も物理的に押下されているものだけ** DOWN（復元）
4. 元のキーの keyup は、押下時に記録したエントリ（VK 別 256 要素の固定配列）に基づき変換する。押下中に修飾キーが変化しても、**押下時に決まったターゲットを解放まで維持**する（リピート中に Ctrl を離しても途中で挙動が変わらない）

## 理由

- 1 回の `SendInput` バッチ送出は、イベント列の間に他の入力が割り込む余地を減らす
- 「まだ押下中の修飾キーのみ復元」は、ユーザーが先に修飾キーを離したケースで修飾キーが押しっぱなしになる（stuck key）事故を防ぐ
- 左右個別追跡は、RCtrl 押下中に LCtrl を解除するような誤補正を防ぐ

## 既知の制限（v0.1 で許容）

- **Alt / Win を含むチョードの置換**では、Alt 単独解放によるメニューバー活性化や Win 単独解放によるスタートメニュー表示が誤発動する可能性がある。Keyhac はダミーキー送出で回避している。発端のユースケース（C-h）では影響しないため v0.1 では対策せず、必要になった時点で ADR を書いて対応する
- `SendInput` が失敗（セキュアデスクトップ等で入力ブロック）した場合は再送しない。フック内でできる安全な回復手段がなく、再送はイベント順序を壊しうるため

## 却下した代替案

- **keydown 時に DOWN+UP のタップを送る**: キーリピートと「押しっぱなし」の意味が失われる（Back 長押しで連続削除できない）
- **押下直後に修飾キーを即復元する**: ターゲットキーの押下中（リピート中）にアプリから見た修飾状態が戻ってしまい、Ctrl+Back 化する
- **`BlockInput` や `keybd_event`**: 前者は権限要件と副作用が大きく、後者は `SendInput` に置き換えられた旧 API
