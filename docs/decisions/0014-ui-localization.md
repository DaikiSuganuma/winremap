# ADR 0014: UI テキストの多言語化（日本語・英語）

- ステータス: 承認（オーナー指示）
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

UI 部分（トレイメニュー、ツールチップ、コンソールの案内文、CLI ヘルプ）を多言語化する。まず日本語と英語。

## 決定

1. **自前の軽量メッセージカタログ**（`src/i18n.rs`）で実装する。静的文字列は `Texts` 構造体（言語ごとに `static`）、引数付きメッセージは言語で `match` する関数
2. 言語選択は起動時に 1 回: `--lang en|ja` があればそれ、なければ **システムの UI ロケール**（`ja*` → 日本語、それ以外 → 英語）。検出には `sys-locale` クレート（MIT OR Apache-2.0、依存極小）を使用し、Win32 API の unsafe 呼び出しを自前で持たない（不変条件 3 の維持）
3. **ローカライズ対象**: トレイメニュー・ツールチップ、起動/リロード時の案内文、CLI ヘルプ、`--debug` の出力
4. **対象外（英語のまま）**: 設定検証エラー（`ConfigError` / `Issue`）と `anyhow` の context 文字列。テストが文言を検証していること、Issue 報告にそのまま貼れることを優先する
5. 新規 UI 文言は英日同時に定義する（03_rust-guidelines.md §11 に明文化）

## 理由

- 文言数が数十件規模のプロジェクトに Fluent / gettext 系のランタイムは過剰。型付きの構造体フィールドなら「片方の言語の定義漏れ」がコンパイルエラーになる
- `sys-locale` により unsafe 隔離を守ったまま OS の言語設定に追従できる

## 却下した代替案

- **fluent / rust-i18n / gettext-rs**: 翻訳ファイルのロード・フォールバック処理などの複雑さが文言規模に見合わない。言語が増えて破綻したら再検討
- **`GetUserDefaultUILanguage` の直接呼び出し**: unsafe を hook/sender/window 以外に置けず、どのモジュールの責務でもない
- **設定エラーメッセージのローカライズ**: 上記 4 のとおり見送り。要望があれば構造化エラー（コード + 引数）への再設計とセットで検討する
