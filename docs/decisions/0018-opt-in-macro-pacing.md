# ADR 0018: オプトインのマクロペーシング（--macro-delay）

- ステータス: 承認
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

差分遷移化（ADR 0017）後もマクロの間欠的な失敗が実機（WinUI 版メモ帳）で報告された。デバッグログの拡充（ADR 0016 追補: マクロ要素列・注入イベントのエコー・外部注入の検出）と並行して、タイミング起因かどうかを利用者側で切り分けられる実験手段が必要。

## 決定

CLI フラグ **`--macro-delay <ms>`（0〜15、既定 0）** を追加する。0 より大きい場合、マクロをストローク単位の `SendInput` に分割し、各ストローク間で指定時間 `sleep` する（Keyhac / AutoHotkey の KeyDelay 相当）。

- **不変条件 2（フックコールバック内のブロッキング禁止）に対する明示的な例外**とし、AGENTS.md に注記する。根拠:
  - 利用者が明示的に指定した場合のみ発動（既定は従来どおり 1 バッチ・sleep なし）
  - 上限 15ms × 最大 8 ストローク = 120ms で、LL フックのタイムアウト閾値（既定数百 ms）を大きく下回る
  - フックコールバック内で待つことで、**マクロ完了前に後続の物理打鍵がアプリに届く順序逆転が起きない**（コールバック実行中はシステムの入力配送が待たされるため）。これがメッセージループ側での遅延実行より優れる点
- ペーシング中は全システムの入力がその時間だけ遅延する。実験・回避用途のフラグであり、既定動作にはしない

## 理由

- 非同期に修飾キーやイベントを処理するアプリに対し、処理猶予を与える古典的で実績のある回避策
- 本体の既定動作（原子的 1 バッチ）の性質を変えずに、利用者が現象を切り分けられる

## 却下した代替案

- **既定でペーシングを有効化**: 大多数のアプリでは不要な入力遅延を常時払うことになる
- **メッセージループ側での遅延実行**: sleep は許されるが、マクロ完了前に後続の物理打鍵が先にアプリへ届く順序逆転が起きる（ADR 0017 で却下済みの理由と同じ）
- **設定ファイル（TOML）での指定**: 実験の反復には CLI の方が速い。恒常設定として必要と分かった時点で config への昇格を再検討する
