# ADR 0025: 表示名を「WinRemap」表記に統一（技術識別子は小文字のまま）

- ステータス: 承認（オーナー指示）
- 日付: 2026-07-19
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

オーナーから「ドキュメント中の winremap が読みにくいので WinRemap にしたい」との提案。命名の由来（ブリーフ §1.4）が挙げる先行例 WinMerge / WinSCP も実際にはキャメルケース表記であり、「WinRemap」の方がむしろ慣習に忠実で、文中の単語境界も見えるため読みやすい。タグ未作成＝初回リリース前であり、変更コストが最小のタイミングでもある。

## 決定

**表示名のみをキャメルケースにする（A 案）。**

1. **「WinRemap」を使う場所**: ドキュメント（README・docs/ 配下）の文章、UI 文字列（トレイのツールチップ、起動・エラーメッセージ、`--help` のタイトル行）
2. **小文字 `winremap` のまま維持する場所（技術識別子）**: Cargo パッケージ名・クレート名、実行ファイル名 `winremap.exe`、設定ディレクトリ `%APPDATA%\winremap\`、GitHub リポジトリ名と URL、名前付きミューテックス等の内部識別子、コマンド実行例、`--version` の出力（コマンド名に一致させる）
3. 過去の ADR（0001〜0024）は書き換えない（decisions/README の原則どおり）
4. 表記ルールを AGENTS.md に明記し、以後の文書作成で統一する

## 理由

- 「読みにくい」という動機はドキュメント上の問題であり、表示名の変更だけで完全に解決する
- Rust エコシステムではパッケージ名は小文字が強い慣習で、「ブランド表記はキャメルケース・パッケージ/バイナリは小文字」の分業は PowerShell/`powershell` など広く前例がある
- 技術識別子に触れないため、ビルド・CI・リリースワークフロー・設定パスへの影響がゼロ

## 却下した代替案

- **B 案: exe 名も `WinRemap.exe` にする**: WinMerge に前例はあるが、Windows はファイル名の大文字小文字を区別しないため実利は表示のみ。ドキュメントのコマンド例・リリースワークフロー・SHA256SUMS の整合コストが増えるだけと判断
- **C 案: GitHub リポジトリ名も変更**: リダイレクトはされるが URL・バッジ・貢献導線の書き換えが必要で、将来 crates.io に公開する場合はどのみちパッケージ名が小文字 `winremap` になり二重管理が残る
