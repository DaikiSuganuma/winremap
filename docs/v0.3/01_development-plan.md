## 2. Phase A — 多言語ファイル化

### 現状

`src/i18n.rs`（629 行）が UI 文字列を全て抱えている。静的文字列は `Texts` 構造体のフィールド（74 個ほど）として英日 2 言語ぶんを Rust の `match` で持ち、書式つきメッセージは関数として並ぶ。言語は起動時に `--lang` かシステムロケールで 1 度だけ決まり（`OnceLock`）、以後変わらない。

この形の問題は 2 つ。

1. **翻訳の追加にソース変更とビルドが要る。** 第三者が言語を足せない
2. **文字列と実装が同じファイルに同居している。** 翻訳者がコードを読む必要がある

### 検討すべき点（結論は調査文書と ADR 0043 へ）

Project Fluent と ICU4X は守備範囲が異なる。両方要るのか、片方で足りるのかを先に決める。

| | 役割 | WinRemap での用途 |
|---|---|---|
| **Project Fluent**（`fluent-bundle`） | メッセージの外部ファイル化（`.ftl`）、複数形・分岐・引数を翻訳側で表現できる | 中心。`Texts` と書式つきメッセージの移行先 |
| **ICU4X**（`icu`） | 数値・日付・時刻・照合順序のロケール対応フォーマット | 現状の用途は日時表示（ログのセッション境界、設定ファイルの更新時刻）程度 |

調査で答えを出す論点:

- **バイナリサイズと依存の増加量。** WinRemap は単一 exe のポータブル配布を売りにしている。ICU4X はロケールデータを伴うため増分が大きくなりうる。ADR 0040（アイコンをビルド時にラスタライズし、ラスタライザ自体は同梱しない）と同じ感覚で、実行時に要るものだけを残せるか
- **翻訳ファイルの置き場所。** exe に埋め込む（第三者は足せないが単一 exe を維持）か、`%APPDATA%\winremap\lang\` 等から実行時に読む（足せるが配布物が増え、壊れたファイルへの耐性が要る）か。両立（埋め込みを既定とし、外部ファイルがあれば優先）も選択肢
- **フォールバック。** 翻訳が欠けたキーは英語へ落とす。現行ガイドライン §11 は「両言語を定義しないとレビューで差し戻し」だが、外部ファイル化すると欠落は実行時に起きうる
- **`--lang` の扱い。** 現在は `en` / `ja` の 2 値。BCP 47 のロケールタグを受ける形へ広げるか
- **技術診断メッセージは英語のまま。** 設定検証エラー・`anyhow` のコンテキストは「issue にそのまま貼れる」ことを優先して英語固定という現行の方針（`src/i18n.rs` 冒頭）を維持するか

参照（公式）:

- Project Fluent: https://projectfluent.org/
- Fluent Syntax Guide: https://projectfluent.org/fluent/guide/
- `fluent-bundle` crate: https://docs.rs/fluent-bundle/
- ICU4X: https://icu4x.unicode.org/
- ICU4X データ管理（`icu4x-datagen`）: https://docs.rs/icu_provider/latest/icu_provider/
- BCP 47 言語タグ: https://www.rfc-editor.org/info/bcp47

### タスク

1. `docs/v0.3/02_i18n-study.md` — 候補比較の調査文書（[v0.2/02_gui-framework-study.md](../v0.2/02_gui-framework-study.md) と同じ体裁）。実測でバイナリサイズ増分を出す
2. ADR 0043 — 多言語化基盤の選定（採用クレート、ファイル形式、埋め込みか外部かの結論）
3. `src/i18n.rs` の移行。`Texts` の各フィールドと書式つきメッセージを翻訳ファイルへ写し、呼び出し側の API はできるだけ変えない（呼び出し箇所が全モジュールに散っているため）
4. 翻訳の網羅性テスト — 全キーが全言語に存在することをテストで担保する（現行の「レビューで差し戻し」を機械化する）
5. 手動確認 — `--lang en` / `--lang ja` / システムロケール自動判定で、トレイ・設定ウィンドウ・ログウィンドウ・ダイアログ・CLI ヘルプの全てが正しく切り替わる

### 完了条件

- UI 文字列が Rust ソースの外にあり、翻訳の追加にコード変更が要らない
- 英語・日本語とも v0.2 と同じ文言が同じ場所に出る（退行がない）
- 翻訳キーの欠落がテストで検出される
- ポータブル exe のサイズ増加が許容範囲に収まっている（許容量は ADR 0043 で決める）

## 3. Phase B — マクロ記憶機能（目玉機能）

### 3.1 オーナー指示（2026-07-21）

- 記憶ボタンを設定ファイルに書く。まずは **`F10`**
- **`S-F10` で記憶開始**、**`S-F10` で記憶終了**（同じキー）
- **`F10` で記憶したマクロを実行**

### 3.2 先に片付ける手続き（ブリーフの改訂が必要）

ブリーフ §3.3 Non-goals に「**キーボードマクロ、スクリプティング**（設定はあくまで宣言的な TOML）」がある。AGENTS.md 禁止事項は「Non-goals は提案のみ可、実装は不可」なので、**実装前にブリーフを改訂する**。

**オーナー決定（2026-07-21）**: IME インジケーター（[ADR 0020](../v0.1/decisions/0020-ime-indicator-scope.md)）と同じ扱いとし、**例外採用**として改訂する。

- Non-goal を「**設定ファイル内のスクリプティング**（設定は宣言的な TOML のまま）」に限定する
- **実行時のキーボードマクロ記録・再生は例外として採用**することをブリーフ §3.3 に明記する
- 判断理由と却下した代替案を **ADR 0044** に残す

### 3.3 不変条件との整合（設計で必ず解く）

| 不変条件 | 論点 | 方針 |
|---|---|---|
| **6. キーロガー化の禁止** | 記憶中は押されたキーをメモリに保持する。形の上ではキーロガー | **記憶内容をディスクに一切書かない**（設定ファイルにも書かない）。**記憶中であることを利用者に見える形で示す**（無自覚に記録が走らない）。プロセス終了で破棄 |
| **2. フックコールバック内の処理制限** | 記憶はコールバック内で発生する。ヒープ確保・ロック待ちは禁止 | 既存のデバッグログ（`DEBUG_RING`、固定長 128、ADR 0016）と同じ**事前確保の固定長リングバッファ**へ積む。例外リストの改訂が要るかは設計時に判断する |
| **1. 自己送出ループの防止** | 再生した `SendInput` のイベントを記憶し直すと無限に育つ | `LLKHF_INJECTED` の素通し規則をそのまま使う。記憶対象からも注入イベントを外す |
| **4. 抑止と送出の順序** | 記憶開始/終了キー（`S-F10`）と再生キー（`F10`）自身は記憶されず、下位アプリにも漏れてはいけない | 記憶ボタンは常に抑止する |

### 3.4 決定済みの仕様（オーナー決定 2026-07-21）

| 論点 | 決定 | 理由 |
|---|---|---|
| **記憶対象** | **リマップ後の出力** | 再生時に既存マクロと同じ送出経路（`sender.rs`）を通せる。物理キーを記憶すると、再生時にリマップを通すのか回避するのかという判断が増え、不変条件 1 との整合も難しくなる |
| **寿命** | **プロセス内のみ**（終了すると消える） | 不変条件 6 との整合が最も取りやすい。Emacs の `kbd-macro` と同じ振る舞いで、利用者の予測も裏切らない |

### 3.5 未決の設計論点（設計書と ADR で詰める）

1. **記憶中であることの提示方法。** トレイアイコンの差し替え（有効/無効の 2 状態アイコンを持つ既存の仕組みが使える）／IME インジケーターと同じ画面上のオーバーレイ／両方。不変条件 6 の「無自覚に記録が走らない」を満たす最小の形を選ぶ
2. **記憶できる長さの上限。** 設定ファイルのマクロは `MAX_MACRO_LEN = 8`（`src/keymap/mod.rs`）だが、記憶マクロは揮発なのでもっと長くてよい。固定長で事前確保する以上、上限は必ず要る。超えたときの挙動（打ち切って通知／記憶を破棄）も決める
3. **記憶スロットの本数。** 1 本（記憶し直すと上書き）か複数か。まずは 1 本で足りるか
4. **設定ファイルの書き方。** 新設の `[macro_record]` セクションか、既存の `[macro]`（現在は `delay_ms` のみ、ADR 0039）へ足すか。キー記法は既存のパーサをそのまま使う
5. **記憶キーと再生キーの関係。** 指示は「`S-F10` で開始/終了、`F10` で再生」で、修飾の有無で役割が変わる。設定に書くのは 1 つのキー（`F10`）で、開始/終了は `Shift` 付きと決め打ちか、開始/終了キーも別に書けるようにするか
6. **再生中の再生・記憶中の再生。** 記憶中に再生キーを押したとき、再生中にさらに再生キーを押したときの扱い
7. **再生時のペーシング。** 既存の `[macro] delay_ms`（ADR 0018/0019）を流用するか、記憶マクロ専用の値を持つか
8. **キーリピート。** 再生キーを押しっぱなしにしたときに繰り返すか（設定ファイルのマクロは自動リピートを抑止している）
9. **設定ウィンドウでの表示。** v0.2 の閲覧専用 GUI に、記憶中の状態や記憶済みマクロの長さを出すか

### タスク

1. ADR 0044 — マクロ記憶機能の採用（ブリーフ §3.3 の例外採用、不変条件 6 との整合方針）
2. ブリーフ §3.3 の改訂（§3.2 のとおり）
3. `docs/v0.3/03_macro-record-design.md` — 設計書。§3.5 の論点に答えを出し、状態遷移（待機 → 記憶中 → 待機、再生中）を明示する
4. 実装 — 記憶リングバッファ（フック側）、状態機械、再生（`sender.rs` の既存経路）、設定ファイルの新項目、記憶中の提示
5. テスト — 設定の読み込みと状態機械は純粋ロジックとしてテスト（`keymap.rs` / `config.rs` の変更にテストを伴わせる規則）。フック層とオーバーレイ描画は手動確認
6. 受け入れチェックリストへの項目追加

### 完了条件

- `S-F10` → 任意のキー操作 → `S-F10` → `F10` で、記憶した操作が再生される
- 記憶中もリマップが停止・遅延しない（不変条件 2 の遵守）
- 記憶中であることが利用者に見えており、記憶内容がディスクに書かれない（不変条件 6 の遵守）
- 記憶ボタン自身が下位アプリへ漏れない
- 上限を超えた記憶が、無言で切り捨てられずに利用者へ伝わる

## 7. フェーズ順序と依存関係

```
Phase A（多言語ファイル化） → Phase B（マクロ記憶） → Phase D（README/サイト） → Phase E（リリース）
                              Phase C（winget/scoop）─────────┘
```

- **Phase A を先行させる。** Phase B と Phase D は新しい UI 文字列を足す。多言語化の基盤を入れ替えた後に書けば、同じ文字列を二度書かずに済む
- **Phase C は他と独立**しており、いつ着手してもよい。ただしマニフェストが参照するのは v0.3.0 のリリース資産なので、**提出は Phase E の後**になる
