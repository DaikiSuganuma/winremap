## 2. Phase A — マクロ記憶機能（目玉機能）

### 2.1 決定済み（[ADR 0043](decisions/0043-runtime-macro-recording.md)、オーナー決定 2026-07-21）

ブリーフ §3.3 の Non-goal「キーボードマクロ、スクリプティング」は **2026-07-21 に改訂済み**。Non-goal を「設定ファイル内のスクリプティング」に限定し、実行時のマクロ記録・再生は IME インジケーター（[ADR 0020](../v0.1/decisions/0020-ime-indicator-scope.md)）と同じく**例外採用**とした。判断理由と却下した代替案は ADR 0043 にある。

| 論点 | 決定 |
|---|---|
| 記憶対象 | **リマップ後の出力**（押した物理キーではなく、WinRemap が変換した結果） |
| 寿命 | **プロセス内のみ**。ディスクにも設定ファイルにも書かない |
| 記憶中の提示 | **記憶中である旨のメッセージを画面に表示し、記憶が終了するまで消さない** |
| 再生中の提示 | **記憶したコマンドを画面に表示する** |
| 記憶できる長さ | **上限 20 コマンド** |
| 記憶スロット | **1 本**（記憶し直すと上書き） |
| 設定の書き方 | **既存の `[macro]` テーブル**（ADR 0039 で新設）に足す |
| 記憶開始 / 終了 / 再生キー | **3 つとも個別に設定できる**。同じキーを開始と終了に書けば「同じキーで開始・終了」になる |
| 再生のペーシング | 既存の `[macro] delay_ms` を流用 |
| 再生中に再生キーを押した場合 | **無視する** |
| キーリピート | **しない**（再生キーを押しっぱなしにしても繰り返さない） |
| 設定ウィンドウでの表示 | 既存のマクロ表示と同様の見せ方に揃える |

オーナーの当初指示にある設定例は次のとおり（`S-F10` で記憶開始、`S-F10` で記憶終了、`F10` で再生）。キー名は設計時に確定する。

```toml
[macro]
delay_ms = 8
# 記憶開始・記憶終了・再生（キー名は設計書で確定）
```

### 2.2 不変条件との整合（設計で必ず解く）

| 不変条件 | 論点 | 方針 |
|---|---|---|
| **6. キーロガー化の禁止** | 記憶中は押されたキーをメモリに保持する。形の上ではキーロガー | ディスクに一切書かない。記憶中は画面表示が出続けるため、無自覚のまま記録が走る状態を作らない。プロセス終了で破棄（ADR 0043 決定 2・3） |
| **2. フックコールバック内の処理制限** | 記憶はコールバック内で起きる。ヒープ確保・ロック待ちは禁止 | デバッグログ（`DEBUG_RING`、固定長 128、ADR 0016）と同じ**固定長・事前確保のバッファ**へ積む。上限 20・スロット 1 本なのでこれが無理なく成立する。例外リストの改訂が要るかは設計時に判断する |
| **1. 自己送出ループの防止** | 再生した `SendInput` のイベントを記憶し直すと無限に育つ | `LLKHF_INJECTED` の素通し規則をそのまま使い、記憶対象からも注入イベントを外す |
| **4. 抑止と送出の順序** | 記憶開始 / 終了 / 再生キー自身は記憶されず、下位アプリへ漏れてもいけない | これらのキーは常に抑止する |

### 2.3 設計（[02_macro-record-design.md](02_macro-record-design.md)、オーナー決定 2026-07-21）

計画時に残っていた論点は設計書で全て解いた。要点は次のとおり。

- **設定は `[macro]` に `record_start` / `record_stop` / `record_play` の 3 項目**。`record_stop` は省略可で、省略すると `record_start` と同じキーになる（「同じキーで開始・終了」が既定形）。3 項目とも省略すれば機能は完全に無効（opt-in）
- **記憶キーを `remap` にも書いた設定は行番号つきエラーで拒否する。** 記憶キーは無条件に最優先で横取りするため、同じ入力の `remap` 規則は決して発火しない「見えない死んだ規則」になる
- **上限 20 に達したら打ち切る**（オーナー決定）。先頭 20 コマンドを確定し、上限に達したことをバナーで知らせる
- **記憶中・再生中のバナーは IME インジケーターとスタイルを似せ、コードは別に持つ**（オーナー決定）。あちらは必ず消えるがこちらは消えてはならず、形も設定項目も違うため。位置はプライマリモニター作業領域の下部中央に固定
- **中断条件**: トレイでの無効化・設定リロードで記憶を中止し破棄する（記憶終了キー自体が設定から来るため、変わると記憶を終われなくなる）

**設計中に見つかった最重要の制約 — 再生をフックコールバック内で行えない。**
記憶マクロは 20 コマンドあり、`delay_ms` が最大の 15ms だと最悪 300ms かかる。これは Windows の低レベルフックのタイムアウト（`LowLevelHooksTimeout`、既定 300ms）に達し、**OS がフックを外して WinRemap のリマップが突然効かなくなる**壊れ方をしうる。メッセージループはフックと同じスレッドなので、そこへ逃がしても解決しない。**再生は専用スレッドで行う**（IME インジケーターと同じ形）。詳細は設計書 §2。

これに伴い**不変条件の改訂が 2 件**必要になる（ADR 0044）。

1. 不変条件 2 の明示的例外に「マクロ記憶スレッドへの `PostThreadMessageW`」を 4 つ目として追加
2. 不変条件 3 の unsafe 隔離リストに `src/macro_record/banner.rs` を追加

## 5. 多言語化を見据えた構造化（v0.3 では移行しない）

多言語ファイル化そのものは見送ったが（§1）、v0.3 は Phase A・C で**新しい UI 文字列を増やす**。将来 Project Fluent / ICU4X へ移す際の作業が「文字列を写すだけ」で済むよう、次を守って書く。

- **UI 文字列は必ず `src/i18n.rs` を通す。** 他のモジュールに直書きしない（現行の規約どおり）
- **キーと値を分けた形を保つ。** `Texts` のフィールド名がそのまま翻訳キーになる想定で命名する。文字列の連結で文を組み立てず、書式つきメッセージは引数を取る関数として 1 箇所にまとめる
- **語順に依存した組み立てをしない。** 「A + 変数 + B」のような英語の語順前提の連結は、翻訳ファイル化したときに翻訳者が直せない。文全体を 1 つのメッセージとして持つ
- **技術診断メッセージは英語のまま。** 設定検証エラー・`anyhow` のコンテキストは「issue にそのまま貼れる」ことを優先するという現行方針（`src/i18n.rs` 冒頭）を維持する

### 将来の検討材料（v0.4 以降）

## 7. フェーズ順序と依存関係

```
Phase A（マクロ記憶） ──┬──▶ Phase C（README/サイト） ──▶ Phase D（リリース）
Phase B（winget/scoop）─┘                                        │
                          （マニフェストの提出はリリース後）◀────┘
```

- **Phase A が本体。** 設計書の承認 → 実装 → 手動受け入れという順で進める
- **Phase B は他と独立**しており、いつ着手してもよい。ただしマニフェストが参照するのは v0.3.0 のリリース資産なので、**提出は Phase D の後**になる
- **Phase C は A と B の結論が出てから**書く
