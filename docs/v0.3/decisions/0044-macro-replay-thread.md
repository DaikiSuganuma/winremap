# ADR 0044: マクロ再生を専用スレッドで行う（不変条件 2・3 の改訂）

- ステータス: 承認（オーナー指示）
- 日付: 2026-07-21
- 作成: Claude Code（AI モデル: claude-opus-4-8）

## 文脈

[ADR 0043](0043-runtime-macro-recording.md) でマクロ記憶機能の採用を決めた。実装方式を [設計書](../02_macro-record-design.md) で詰める過程で、**再生をフックコールバック内で行えない**ことが分かった。

既存の設定ファイル由来のマクロ（`Output::Seq`）は、フックコールバックの中で `sender::send_sequence` を同期的に呼んでいる（`src/hook.rs` `emit_output`）。要素間に `[macro] delay_ms` の sleep が入るため、不変条件 2 は「利用者が `--macro-delay` を指定した場合の有界な sleep（**上限 15ms × 8**、[ADR 0018](../../v0.1/decisions/0018-opt-in-macro-pacing.md)）」を明示的な例外として認めている。8 は `MAX_MACRO_LEN` に由来し、最悪 120ms である。

記憶マクロの上限は **20 コマンド**（ADR 0043 決定 5）なので、同じやり方だと最悪 **15ms × 20 = 300ms** かかる。

Windows は低レベルフックのコールバックにタイムアウトを設けている。`HKEY_CURRENT_USER\Control Panel\Desktop\LowLevelHooksTimeout`（既定 **300ms**）を超えると、OS はそのフックの応答を待たずにイベントを配送し、フックを事実上外す。つまり **「マクロを再生すると WinRemap のリマップが突然効かなくなる」**という壊れ方をしうる。

メッセージループへ逃がしても解決しない。メッセージループはフックと同じスレッドで回っており（`hook.rs` `run_message_loop`）、そこで 300ms 費やせばフックコールバックの処理も同じだけ遅れる。

参照（公式）:

- LowLevelKeyboardProc（フックがタイムアウトすると呼ばれなくなる旨の記述）: https://learn.microsoft.com/en-us/windows/win32/winmsg/lowlevelkeyboardproc
- SetWindowsHookExW: https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-setwindowshookexw
- SendInput: https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-sendinput

## 決定

1. **記憶マクロの再生は専用スレッド（マクロ記憶スレッド）で行う。** フックコールバックからは有界・非ブロッキングな `PostThreadMessageW` を 1 回投げるだけとする。IME インジケーター（[ADR 0020](../../v0.1/decisions/0020-ime-indicator-scope.md)）が取った形をそのまま踏襲する

2. **同じスレッドが記憶中・再生中バナーのウィンドウを持つ。** ウィンドウのメッセージは作成したスレッドが回す必要があるため、再生と同居させてスレッドを 1 本に収める

3. **不変条件 2 の明示的例外に 4 つ目を追加する。** 「マクロ記憶スレッドへの `PostThreadMessageW`（本 ADR）」。AGENTS.md 不変条件 2 とブリーフ §5-2 を改訂する

4. **不変条件 3 の unsafe 隔離リストに `src/macro_record/banner.rs` を追加する。** AGENTS.md 不変条件 3 とブリーフ §5-3 を改訂する。`src/macro_record/` の他のファイル（`mod.rs` / `recorder.rs`）は unsafe を含まない

5. **既存の設定マクロはコールバック内のままとする。** 経路の統合は v0.3 では行わない

## 理由

- **迷ったときの優先順位は「安定性（フックを止めない） > 単純さ > 機能の豊富さ」である。** フックが OS に外されるのは、この優先順位が守ろうとしているものそのものの喪失にあたる。ここで単純さを取る理由はない
- **前例がそのまま使える。** IME インジケーターで「重い仕事は専用スレッド、フックからは有界・非ブロッキングな `PostThreadMessageW` のみ」という形は既に確立しており、失敗隔離（`catch_unwind` でスレッド内に閉じ込め、フックは動き続ける）まで含めて踏襲できる。新しい構造を発明していない
- **送出元スレッドが変わっても挙動は変わらない。** `SendInput` はどのスレッドから呼んでもよく、注入イベントは `LLKHF_INJECTED` によりフックで素通しされる（不変条件 1）。既存の `sender::send_sequence` をそのまま呼べる
- **例外 4 は例外 3 と同じ性質である。** どちらも「フックは投げるだけ、受け手が専用スレッド」であり、コールバック内の処理量は変わらない。既存の例外リストの考え方に新しい種類のリスクを持ち込んでいない
- **バナーの unsafe は既存の隔離方針の範囲に収まる。** レイヤードウィンドウの生成・描画は Win32 そのものであり、`src/ime_indicator/overlay.rs` や `src/gui/win32.rs` と同じ扱いで 1 ファイルに閉じる。`macro_record` の他 2 ファイルは unsafe を含まないため、機能全体が unsafe になるわけではない
- **既存の設定マクロを触らない理由**は、8 要素・最大 120ms で実績があり、それを目玉機能と同時に作り替えると、不具合が出たときにどちらが原因か切り分けられなくなるため。統合すれば例外 2 を消せるので、将来の整理候補として設計書に記録する

## 却下した代替案

- **再生をフックコールバック内で行い、例外 2 の上限を 15ms × 20 に広げる**: 実装は最も小さいが、最悪 300ms は `LowLevelHooksTimeout` の既定値そのものである。境界ぴったりに設計を寄せることになり、しかも失敗したときの症状は「リマップが黙って効かなくなる」という最も分かりにくい形になる。不変条件 2 が存在する理由（Keyhac のタイムアウト問題を再現しない、ブリーフ §1.3）に正面から反する

- **再生をメッセージループへ逃がす**: 一見「コールバックの外」に見えるが、メッセージループはフックと同じスレッドで回っている。300ms 占有すればフックの処理も同じだけ遅れるため、何も解決しない。本 ADR で最も注意すべき誤りであり、明示的に却下として残す

- **記憶マクロの上限を 8 に下げて既存の例外に収める**: 例外を増やさずに済むが、オーナーが決めた上限 20（ADR 0043）を実装の都合で削ることになる。制約は機能側ではなく実装側にあり、専用スレッドで解けるのだから、機能を削る理由がない

- **`delay_ms` を記憶マクロでは無視して常にバースト送出する**: sleep が無くなればコールバック内でも短時間で終わるが、`delay_ms` はバースト注入を取りこぼすアプリのために入れた設定である（ADR 0018）。記憶マクロだけそれを無視すれば、まさにそのアプリで再生が壊れる

- **記憶マクロ専用の送出関数を書く**: `sender::send_sequence` を使わず軽量版を作る案。修飾キーの持ち上げ・復帰（[ADR 0019](../../v0.1/decisions/0019-macro-delay-in-config.md)）やマーカー付与といった、時間をかけて詰めた挙動を二重に持つことになる。設定マクロと記憶マクロで再生結果が違うのは利用者にとっても不可解であり、同じ経路を通す

- **バナーを既存の `src/ime_indicator/overlay.rs` に相乗りさせる**: unsafe の追加ファイルを増やさずに済むが、あちらは `TIMER_HOLD` → `TIMER_FADE` で必ず消える作りで、消えてはならないバナー（ADR 0043 決定 3）とは要件が逆である。実績のあるコードに「消える版 / 消えない版」の分岐を全体へ差し込むより、100 行程度を別ファイルに持つほうが安全（オーナー決定 2026-07-21）
