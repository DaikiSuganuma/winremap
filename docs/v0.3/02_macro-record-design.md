## 1. 何を作るか

記憶開始キーを押してから記憶終了キーを押すまでの操作を覚え、再生キーでその操作を送出する。Emacs の `start-kbd-macro` / `end-kbd-macro` / `call-last-kbd-macro` に相当する。

```
[S-F10]  →  C-a  C-k  C-y  ...  →  [S-F10]  →  [F10] [F10] [F10]
記憶開始      この間の操作を覚える      記憶終了      何度でも再生
```

ADR 0043 で決まっていることは前提として扱う（記憶するのはリマップ後の出力、寿命はプロセス内のみ、ディスクに書かない、記憶中は画面に出続ける、上限 20 コマンド・スロット 1 本、設定は `[macro]`、ペーシングは `delay_ms` 流用、再生中の再生キーは無視、キーリピートしない）。

**「コマンド」＝ 1 つのキーコンビネーション（chord）** を指す。`C-a` で 1 コマンド、`C-x` `C-s` の 2 ストロークは 2 コマンドである。

## 2. 設計を決める前に見つかった制約 — 再生をフックコールバック内で行えない

**これが本設計で最も重要な判断である。**

既存の設定ファイル由来のマクロ（`Output::Seq`）は、フックコールバックの中で `sender::send_sequence` を同期的に呼んでいる（`src/hook.rs` `emit_output`）。要素間に `delay_ms` の sleep が入るため、不変条件 2 は「利用者が `--macro-delay` を指定した場合の有界な sleep（**上限 15ms × 8**、[ADR 0018](../v0.1/decisions/0018-opt-in-macro-pacing.md)）」を明示的な例外として認めている。8 は `MAX_MACRO_LEN` に由来し、最悪 120ms である。

記憶マクロは **20 コマンド**なので、同じやり方だと最悪 **15ms × 20 = 300ms** になる。これは看過できない。

- Windows は低レベルフックのコールバックに**タイムアウト**を設けている（`HKEY_CURRENT_USER\Control Panel\Desktop\LowLevelHooksTimeout`、既定 **300ms**）。超えると **OS がそのフックを黙って外す / 以後のイベントを待たずに配送する**
- つまり「再生すると WinRemap のリマップが突然効かなくなる」という壊れ方をしうる
- 迷ったときの優先順位は **安定性（フックを止めない） > 単純さ > 機能の豊富さ** である。ここで単純さを取る理由はない

**メッセージループへ逃がすだけでは解決しない。** メッセージループはフックと同じスレッドで回っており（`hook.rs` `run_message_loop`）、そこで 300ms 費やせばフックコールバックの処理も同じだけ遅れる。

**決定: 再生は専用スレッドで行う。** IME インジケーターが取った形（ADR 0020: 重い仕事は専用スレッド、フックからは有界・非ブロッキングな `PostThreadMessageW` のみ）をそのまま踏襲する。`SendInput` はどのスレッドから呼んでもよく、注入イベントは `LLKHF_INJECTED` で素通しされる（不変条件 1）ため、送出元スレッドが変わっても既存の挙動は変わらない。

これに伴い**不変条件の改訂が 2 件**必要になる（→ [ADR 0044](#8-必要な-adr)）。

1. 不変条件 2 の明示的例外に「マクロ記憶スレッドへの `PostThreadMessageW`」を **4 つ目**として追加する
2. 不変条件 3 の unsafe 隔離リストに `src/macro_record/banner.rs` を追加する

なお、既存の設定マクロ（8 要素・最大 120ms）はコールバック内のままとする。本 v0.3 の範囲で経路を統合はしない（統合すれば例外 2 を消せるが、実績のある経路を目玉機能と同時に触ることになる）。将来の整理候補として記録する。

## 3. モジュール構成

IME インジケーターと同じ「独立機能としてディレクトリに隔離し、既存コードへの接点を最小に保つ」形をとる。

```
src/macro_record/
  mod.rs      — オーケストレーション。unsafe を含まない（不変条件 3）
  recorder.rs — 記憶バッファと状態機械。純粋ロジック（テスト対象）
  banner.rs   — 記憶中・再生中バナーのウィンドウと Win32。unsafe はここだけ
```

`recorder.rs` の `Recorder`（状態 + 固定長配列）は **`winremap` ライブラリクレット側**（`src/lib.rs` から公開）に置く。`keymap` / `config` と同じく、バイナリを起動せずにテストできるようにするため。

既存コードへの接点:

| ファイル | 接点 |
|---|---|
| `hook.rs` | キーダウン処理に「記憶キーの判定」と「記憶への push」を各 1 箇所 |
| `window.rs` | フォアグラウンド変更時の通知 1 行（バナーの追従用、§6.2） |
| `main.rs` | 起動時の `sync_with_config()`、終了時の `stop()` |
| `tray.rs` | 有効/無効トグル時とリロード時の通知 |
| `config/` | `[macro]` の 3 項目の読み込みと検証 |
| `gui/config_window.rs` | `[macro]` セクションへの 3 行追加と、記憶済みマクロの表示 |
| `i18n.rs` | 新規 UI 文字列（計画 §5 の構造化ルールに従う） |

## 4. 設定ファイル

既存の `[macro]` テーブル（[ADR 0039](../v0.2/decisions/0039-macro-section-in-config.md)）に 3 項目を足す。

```toml
[macro]
delay_ms = 8          # 既存。再生のペーシングにも流用する

record_start = "S-F10"  # 記憶開始
record_stop  = "S-F10"  # 記憶終了（省略時は record_start と同じ）
record_play  = "F10"    # 再生
```

- 値はいずれも既存のキー記法（`src/keymap/parse.rs`）。パーサをそのまま使う
- **`record_stop` は省略可**で、省略時は `record_start` と同じキーになる。オーナー指示の「同じキーで開始・終了」がこの既定形にあたる。別のキーを書けば別々になる
- **3 項目とも省略すれば機能は完全に無効**（opt-in、ADR 0043 決定 7）。スレッドもウィンドウも作らない

### 検証（保存前・行番号つき）

| 条件 | 扱い |
|---|---|
| キー記法として解釈できない | エラー（既存の記法エラーと同じ文言体系） |
| 修飾キー単体（`Ctrl` など）を指定 | エラー（入力として修飾キー単体を禁じる既存規則と同じ） |
| `record_start` だけ、または `record_play` だけを書いた | エラー。「`record_start` と `record_play` の両方が必要」 |
| `record_play` が `record_start` または `record_stop` と同じ | エラー。同じキーが再生と記憶開始を兼ねると、どちらの意味か決められない |
| `record_start` == `record_stop` | **正常**（トグル動作。既定形） |
| いずれかのキーが `[[keymap]]` の `remap` に書かれている | **エラー** |

最後の 1 件を警告ではなくエラーにする理由: 記憶キーは全アプリで無条件に最優先で横取りされる（§5.1）。同じ入力を `remap` に書いても**決して発火しない死んだ規則**になり、しかも設定ファイルを読んでもそうとは分からない。キーマップ同士の重複は「アプリ別がグローバルに勝つ」という意味のある優先順位なので設定ウィンドウで見せるに留めているが（v0.2）、こちらは意味のある優先順位ではなく単なる不可視の無効化なので、書いた時点で止める。

判定は、`remap` の完全一致規則（`exact`）に同じコンビネーションがある場合と、素キー規則（`bare`）に同じ VK がある場合の両方。素キー規則は修飾を無視して一致するため、`F10 = "Home"` は `record_play = "F10"` とも `record_start = "S-F10"` とも衝突する。

## 5. 動作

### 5.1 記憶キーの横取り位置

`hook.rs` の `on_key_down` で、**自動リピート判定とトレイの有効/無効判定の後、キーマップ解決の前**に記憶キーを判定する。

```
on_key_down(vk, physical_repeat)
  ├ ACTIVE[vk] がある → 自動リピート処理（記憶キーもここで止まる = リピートしない）
  ├ !ENABLED         → 素通し（無効時は記憶機能も動かない）
  ├ ★ 記憶キーの判定 ← ここ
  ├ PENDING（2 ストローク目）の解決
  └ 通常のキーマップ解決
```

記憶キーに一致したら、対応する処理を行ったうえで **`ACTIVE[vk] = SuppressUp` を立てて `true` を返す**（イベントを抑止する）。これにより:

- 記憶キー自身が下位アプリへ漏れない（不変条件 4）
- キーアップも抑止され、アプリが「押されたまま」と誤認しない
- 押しっぱなしは自動リピート分岐で `SuppressUp` に落ち、何も起きない（オーナー決定「キーリピートはしない」）
- 修飾に Alt / Win を含む場合は既存の `arm_menu_guard` を通し、メニューバー / スタートメニューが開かないようにする（[ADR 0015](../v0.1/decisions/0015-menu-mask-key.md) の仕組みをそのまま使う）

### 5.2 状態機械

状態はフックスレッドの `thread_local` に置く（すべての遷移がフックスレッド上で起きるため、ロックが要らない）。

```
                record_start
      ┌──────────────────────────────┐
      │                              ▼
  ┌────────┐                   ┌───────────┐
  │ Idle   │◀── record_stop ───│ Recording │──┐
  └────────┘                   └───────────┘  │ 21 コマンド目
      │  ▲                           ▲        │（打ち切り）
      │  │                           └────────┘
      │  │ 再生完了
      │  │
      └──┴── record_play（記憶済みマクロがある場合のみ）→ 再生スレッドへ post
```

| 状態 | 入力 | 動作 |
|---|---|---|
| Idle | `record_start` | Recording へ。バッファを空にし、バナーを「記憶中 0/20」で表示 |
| Idle | `record_play` | 記憶済みマクロがあれば再生スレッドへ post。無ければ何もしない（デバッグログに 1 行） |
| Idle | `record_stop` | 何もしない |
| Recording | 記憶対象のキー | バッファへ push、バナーの件数を更新 |
| Recording | `record_stop` | Idle へ。バッファを確定し、バナーを消す |
| Recording | `record_play` | **無視する**（入れ子の再生は記憶内容の意味を曖昧にする）。デバッグログに 1 行 |
| Recording | 21 コマンド目 | **打ち切る**（オーナー決定）。先頭 20 コマンドをマクロとして確定し、Idle へ。バナーで上限到達を知らせてから消す |
| 再生中 | `record_play` | **無視する**（オーナー決定。§5.5） |

### 5.3 何が記憶されるか

記憶するのは**リマップ後の出力**（ADR 0043 決定 4）。`on_key_down` が解決した結果に応じて次を push する。

| 解決結果 | 記憶する内容 |
|---|---|
| `Resolution::Exact(Output::Chord(target))` | `target` を 1 コマンド |
| `Resolution::Exact(Output::Seq(seq))` | `seq` の各要素を 1 コマンドずつ（マクロを記憶すると展開される） |
| `Resolution::KeyOnly(target_vk)` | `{ 現在押されている修飾, target_vk }` を 1 コマンド。素キー置換は修飾に触れないため、押していた修飾がそのまま乗る |
| `Resolution::Prefix` | 何も記憶しない（プレフィックス単体は何も出力していない）。2 ストローク目の出力が記憶される |
| 2 ストローク目が未定義（swallow） | 何も記憶しない（何も出力していない） |
| 解決なし（素通し） | 押されたコンビネーションをそのまま 1 コマンド |
| 注入イベント（`LLKHF_INJECTED`） | 何も記憶しない。`handle_event` の先頭で return するため、そもそも到達しない（不変条件 1 と同じ守り） |
| 修飾キー単体の押下 | 何も記憶しない。修飾は各コマンドの一部として乗る |

**この設計の限界（README に書く）**: 記憶されるのは「どのコマンドを、どの順で打ったか」であって、**キーを押していた長さや、押しっぱなしにしながら別のキーを押したこと**ではない。再生は各コマンドを 1 回ずつ叩く。Emacs のキーボードマクロも同じ性質を持つ。ゲームのような同時押しの再現には使えない。

### 5.4 記憶バッファ（フック安全）

```rust
pub const MAX_RECORDED_LEN: usize = 20;

pub struct Recorder {
    state: RecordState,                       // Idle / Recording
    buffer: [KeyCombo; MAX_RECORDED_LEN],     // 事前確保・固定長
    len: usize,
}
```

- `hook.rs` の `DEBUG_RING`（固定長 128、[ADR 0016](../v0.1/decisions/0016-debug-key-logging.md)）と同じ形。**コールバック内でのヒープ確保もロック待ちも発生しない**（不変条件 2）
- `KeyCombo` は `Copy` なので、配列の初期化も push もコピーだけで済む
- 20 コマンド × 数バイトなので、常時確保しておくコストは無視できる

### 5.5 確定と再生

記憶バッファはフックスレッドの `thread_local` にあるため、再生スレッドからは読めない。**確定したマクロだけを共有領域へ写す**。

```
[フックコールバック]                     [メッセージループ（同スレッド）]
record_stop / 上限到達
  → 状態を Idle に
  → 自スレッドへ PostThreadMessageW ──▶ thread_local からバッファを読み、
     （--debug のログと同じ手法）        RECORDED（ArcSwapOption<Vec<KeyCombo>>）へ
                                          publish。確保はここで行う（安全）

[フックコールバック]                     [マクロ記憶スレッド]
record_play
  → REPLAYING が true なら無視
  → 押されている修飾を wparam に載せて
    PostThreadMessageW ────────────────▶ REPLAYING を立てる
                                          RECORDED を load
                                          バナーを「再生中: …」にして即座に再描画
                                          sender::send_sequence（delay_ms 込み）
                                          バナーを消す
                                          REPLAYING を下ろす
```

- `RECORDED` は `ArcSwapOption<Vec<KeyCombo>>`。`REMAP_TABLE` と同じ待機なしの読み取りで、`arc-swap` は既に依存にある
- `REPLAYING` は `AtomicBool`。フックからの読み取りは待機なしのロードだけで、これが「再生中に再生キーを押したら無視」を成立させる
- 再生スレッドへの post は**有界・非ブロッキング**（`PostThreadMessageW` は失敗しても捨てるだけ）。不変条件 2 の新しい例外 4 にあたる
- 再生の送出は既存の `sender::send_sequence` をそのまま使う。設定マクロと完全に同じ経路・同じ `delay_ms`

### 5.6 中断される条件

| 出来事 | 記憶中の扱い | 記憶済みマクロの扱い |
|---|---|---|
| トレイで無効化 | **中止して破棄**。バナーを消す | 保持する（再度有効化して再生できる） |
| 設定のリロード | **中止して破棄**。バナーを消す | 保持する |
| WinRemap 終了 | 破棄 | 破棄（ADR 0043 決定 2） |

無効化とリロードで記憶を破棄する理由: 記憶キーそのものが設定から来るため、リロードで記憶終了キーが変わったり消えたりすると、**記憶を終われなくなる**。無効化も同様に、`ENABLED` が false の間は記憶キーが横取りされない（§5.1）ので終了できない。終われない記憶が画面にバナーを出したまま残るのが最悪なので、どちらも中止する。破棄されたことはバナーとデバッグログで伝える。

## 6. 記憶中・再生中の画面表示（バナー）

### 6.1 IME インジケーターとの関係

**スタイルは似せ、コードは別に持つ**（オーナー決定 2026-07-21）。`src/macro_record/banner.rs` を新設する。

似せる点（IME インジケーターと同じ Win32 の作り）:

- レイヤードウィンドウ + 角丸リージョン、暗い背景に白い文字、半透明
- `WS_EX_TRANSPARENT`（クリックが下へ抜ける）、`WS_EX_NOACTIVATE`（フォーカスを奪わない）、`WS_EX_TOPMOST`、`WS_EX_TOOLWINDOW`（タスクバー / Alt-Tab に出ない）、`WS_EX_LAYERED`
- フォントは Yu Gothic UI

別のコードにする理由:

- **消えない**。IME インジケーターは「今この瞬間の状態」を伝えるものなので、`TIMER_HOLD` → `TIMER_FADE` で必ず消える。マクロ記憶のバナーは記憶が終わるまで消えてはならず（ADR 0043 決定 3）、タイマー処理・フェードのアルファ制御・`FADE_ALPHA` の状態がまるごと不要になる
- **形が違う**。あちらは 1 文字の正方形。こちらは「記憶中 3/20」や「再生中: C-a → C-k → C-y」という**可変長の横長テキスト**で、幅の決め方も描画も別物
- **設定項目が違う**。`IndicatorSettings`（`duration_ms` / `size` / `opacity` / `trigger_keys` / `show_app_name`）はどれも当てはまらない
- 共通化すると、消える版と消えない版の分岐が `overlay.rs` の全体に入り込む。実績のあるコードに条件分岐を差し込むより、100 行程度を別に持つほうが安全

### 6.2 表示位置（オーナー決定により改訂 2026-07-21）

**前面ウィンドウがあるモニターの作業領域の下部中央**に表示する。フォーカスが別のモニターへ移ったら、バナーもそのモニターへ移る。

当初はプライマリモニター固定とし、「記憶はアプリをまたぐのでフォアグラウンドを追う必然性がない」「追跡と移動処理が増える」を理由としていた。**実機確認でこれが誤りと分かった**（オーナー指摘）。複数モニター環境では、作業しているモニターと別の場所にバナーが出るため**気づかない**。バナーは「無自覚のまま記録が走らないこと」（ADR 0043 決定 3）を担保する唯一の手段であり、気づかれないバナーはその役目を果たさない。不変条件 6 との整合を表示に依存させている以上、見えないことは実装上の欠陥にあたる。

- 追跡は既存の `SetWinEventHook`（`src/window.rs`）に接点を 1 行足すだけで済む。IME インジケーターが使っているのと同じ経路
- 移動先モニターは `MonitorFromWindow` + `GetMonitorInfoW` の `rcWork`（タスクバーを除いた作業領域）。前面ウィンドウが無い場合はプライマリモニターにフォールバックする
- 画面中央に出し続けると作業の邪魔になるため、下部中央である点は変えない

### 6.3 表示内容

| 状態 | 内容 |
|---|---|
| 記憶中 | 「マクロ記憶中  3/20   notepad.exe で記憶中  —  S-F10 で終了 / F10 で再生」 |
| 上限到達 | 上限に達して記憶を終了した旨を数秒表示してから消す |
| 中断（無効化・リロード） | 中断した旨を数秒表示してから消す |
| 再生中 | 「notepad.exe で再生中:  C-a → C-k → C-y」— 記憶したコマンドを既存のマクロ表示と同じ ` → ` 区切りで並べる（`gui/config_window.rs` の `Output::Seq` の表示に揃える）。長い場合は末尾を省略記号にする（`DT_END_ELLIPSIS`） |

記憶中の行に含める要素と、その理由（オーナー決定 2026-07-21）:

- **件数 `3/20`** — 上限が見えるので、打ち切りが不意打ちにならない
- **記憶中のアプリ名** — 打鍵がどのアプリへ入っているかを示す。アプリを切り替えれば表示も切り替わる。ウィンドウタイトルではなく exe 名を出す（タイトルは文書名など機微な情報を含みうる。[ADR 0024](../v0.1/decisions/0024-ime-indicator-app-name.md) と同じ判断）
- **記憶終了キーと再生キー** — **記憶を終われなくなることが、この機能の最悪の失敗**である。終了キーを設定ファイルで確認しに行かせない。記憶中は必ずこの 2 つが見えている状態を保つ

アプリ名が取得できない場合（管理者権限ウィンドウは UIPI で照会を拒否する。ただしそのウィンドウには注入入力も届かない — ブリーフ §5-5）は「不明なアプリ」と表示する。

文字列は全て `i18n.rs` を通す（計画 §5）。アプリ名やキー名を差し込んだ文であっても、**文全体を 1 つの書式文字列**として持つ（計画 §5 の語順ルール）。

アプリ名とキーはメッセージループではなく**マクロ記憶スレッド側で描画時に解決する**。バナーコマンドが運ぶのは件数だけとし、記憶中にフォーカスが移っても表示が古くならないようにするため。

### 6.4 スレッド

バナーウィンドウと再生を**同じ 1 本のスレッド**が持つ。

- ウィンドウは作成したスレッドがメッセージを回す必要があるため、バナーには専用スレッドが要る。再生も同じスレッドで行えば、スレッドは 1 本で済む
- 再生中はそのスレッドが sleep を挟んで送出するのでメッセージを回せない。そのため**再生を始める前にバナーの内容を確定し、同期的に再描画してから**送出に入る（`InvalidateRect` + `UpdateWindow`）。再生は最大 300ms で、その間バナーが更新されないだけなので実害はない
- スレッドは記憶機能が設定で有効なときだけ起動する。パニックは `catch_unwind` でこのスレッドに閉じ込め、フックは動き続ける（IME インジケーターと同じ失敗隔離）

## 7. 設定ウィンドウでの表示

既存の `[macro]` セクション表示（`gui/config_window.rs` の `macro-settings` テーブル）に行を足す。

| 項目 | 値 |
|---|---|
| 記憶開始 | `S-F10` |
| 記憶終了 | `S-F10` |
| 再生 | `F10` |
| 送出間隔 | `8 ms`（既存） |
| **記憶したマクロ** | `C-a → C-k → C-y`（未記憶なら「なし」） |

- 設定ファイルに書かれたコメントを添える既存の仕組み（`comments.general("macro.record_start")` 等）をそのまま使う
- 「記憶したマクロ」だけは設定ファイル由来ではなく実行時の状態である。**ディスクには存在しない内容を表示している**ことが分かるよう、他の行と区別できる見せ方にする（設定ウィンドウは v0.3 でも閲覧専用なので、編集はできない）
- コマンドの並べ方は既存のマクロ表示と同じ ` → ` 区切り（オーナー決定）

## 11. 受け入れ確認に入れる項目（抜粋）

- 記憶 → 終了 → 再生が動く。再生を複数回繰り返しても同じ結果になる
- 記憶キー（開始・終了・再生）が下位アプリへ漏れない。テキストエディタで文字が入らない、Alt 付きにしてもメニューバーが開かない
- 記憶中もリマップが停止・遅延しない（不変条件 2）
- **再生中にリマップが止まらない**（§2 の本題。`delay_ms` を最大の 15 に設定し、20 コマンドを記憶して再生しながら別のキーを打つ）
- 記憶中バナーが記憶終了まで消えない。クリックが下へ抜ける。フォーカスを奪わない
- 21 コマンド目で打ち切られ、その旨が表示される
- 再生中に再生キーを押しても二重に走らない
- 記憶中にリロード / 無効化すると中止され、バナーが消える
- 記憶キーを `remap` にも書いた設定が、行番号つきエラーで拒否される
- WinRemap を再起動すると記憶が消えている（ディスクに残っていない）
