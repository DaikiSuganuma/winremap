# ADR 0004: キーマップのマッチング意味論と優先順位

- ステータス: 承認
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

複数の `[[keymap]]` セクション（グローバル `*` とアプリ別）が同じキーにルールを持つ場合の優先順位と、修飾キーの一致判定の意味論を決める必要がある。仕様の詳細は [04_config-spec.md](../02_config-spec.md) §3。

## 決定

1. **ルールを 2 種類に分ける**
   - 修飾キー付きルール（`"C-h" = "Back"`）: 修飾キー状態の**完全一致**で判定（`C-h` は Ctrl+Shift+H にマッチしない）。置換時は修飾キー状態も RHS に合わせる
   - 単キールール（`"CapsLock" = "LCtrl"`）: 修飾キー状態を**無視**してキーだけ差し替える。物理修飾キーには触らない。RHS への修飾キー指定は v0.1 ではエラー
2. **優先順位**: アプリ別キーマップ > グローバル（定義順に関係なく）→ 同分類内は定義順の先勝ち → 同一キーマップ内は修飾キー付き > 単キー
3. 正規化後に同一キーマップ内で重複するルール（`"C-h"` と `"c-H"` 等）は設定エラーとして一括報告する

## 理由

- 完全一致は xremap の keymap と同じ意味論で、「`C-h` を定義したら Ctrl+Shift+H まで暴発する」事故を防ぐ。単キールールの修飾無視は CapsLock→Ctrl のような「キー差し替え」用途（xremap の modmap 相当）に必要で、両者は目的が異なる
- 「アプリ別が常に勝つ」は、グローバル定義を後から書き足してもアプリ別の意図が壊れない、設定ファイルの並び順に依存しない、という予測可能性のため
- 先勝ち（first-match）は「上に書いたものが優先」という xremap と同じ直感に合わせた

## 却下した代替案

- **後勝ち（CSS 的な上書き）**: 「後から読み込んだものが勝つ」は複数ファイル結合がある場合の慣習。単一ファイルの v0.1 では、定義が離れた場所にあるほど挙動が読みにくくなるため却下
- **部分一致（`C-h` が Ctrl+Shift+H にもマッチ）**: 暴発の温床。Shift 込みで反応させたいなら `"C-S-h"` を明示的に書けばよい
- **全キーマップの辞書を起動時に 1 つへマージ**: ルックアップは速くなるが、アプリごとの適用セットが動的（前面ウィンドウ依存）なので、結局アプリ別の構造が必要。キーマップ配列の線形走査は設定数規模（高々数十）では十分速い
