# ADR 0003: 設定リロードは arc-swap による atomic swap で行う

- ステータス: 承認
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

フックコールバック内ではロック待ちが禁止されている（AGENTS.md 不変条件 2。Keyhac のフックタイムアウト問題を再現しないため）。一方で設定のホットリロード（Phase 3）では、動作中に設定データを差し替える必要がある。コールバックが参照する設定データの共有方法を決める。

## 決定

- 設定は読み込み時に読み取り専用の `RemapTable`（ルックアップ最適化済み構造）へコンパイルし、**`arc_swap::ArcSwap<RemapTable>` で共有する**
- フック側は `load()`（wait-free）で参照するのみ。リロードはフック外のスレッドで新しい `RemapTable` を構築してから `store()` で一括差し替える
- クレート追加は実際に共有が発生する Phase 2 で行う（Phase 1 の純粋ロジックには不要のため、未使用依存を持たない）。arc-swap のライセンスは MIT/Apache-2.0 デュアルで方針に適合

## 理由

- `ArcSwap::load` はロックフリーかつほぼ wait-free で、コールバック内の「ロック待ち禁止」制約を満たす
- 「構築してから一括差し替え」により、リロード中のキーイベントは旧設定か新設定のどちらか一貫した状態だけを見る（部分更新状態が存在しない）
- 実績のある定番クレートで、自前実装より検証コストが低い

## 却下した代替案

- **`Mutex` / `RwLock`**: リロード書き込み中にコールバックがブロックされ、不変条件 2 に違反する
- **`std::sync::atomic` + `Arc` の自前 swap**: `AtomicPtr` ベースの手書きは参照カウントの解放タイミングの扱いを誤りやすい。arc-swap はまさにこの問題の解決策として作られている
- **チャネルでフックスレッドに新設定を送る**: フック側に受信処理（分岐・所有権移動）が増え、コールバックを薄く保つ方針に反する
