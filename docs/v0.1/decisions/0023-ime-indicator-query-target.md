# ADR 0023: インジケーターの照会対象ウィンドウの解決とシェル面の抑制

- ステータス: 承認（オーナーからの改善要望への対処）。決定 1 の「状態更新も行わない」は [ADR 0026](0026-ime-indicator-reshow-after-shell.md) で一部上書き（2026-07-19: シェル面経由で戻ったアプリには再表示する）
- 日付: 2026-07-19
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

Phase I4 の実使用でオーナーから 2 件の報告があった。

1. **タスクバーをクリックするとパネルが表示される**: タスクバーは explorer.exe のウィンドウ（クラス `Shell_TrayWnd`）で、クリックのたびにフォアグラウンド変化として検知される。システムの IME 状態がオンだとタスクバー中央にパネルが出てしまう
2. **設定アプリの「設定の検索」でパネルが出ない**: UWP 系アプリでは前面ウィンドウが `ApplicationFrameHost.exe` のフレームウィンドウであり、実際の入力先は**別プロセスが所有する子の `Windows.UI.Core.CoreWindow`**。フレームウィンドウの既定 IME ウィンドウへ `IMC_GETOPENSTATUS` を送っても常にオフが返る（デバッグログで確認）

## 決定

1. **シェル面の無視**: 前面ウィンドウのクラス名が `Shell_TrayWnd` / `Shell_SecondaryTrayWnd`（タスクバー）、`Progman` / `WorkerW`（デスクトップ）の場合、照会・表示・非表示・状態更新のすべてを行わない。タスクバーをクリックして元のアプリに戻っても「オフ→オン遷移」と誤認せず、パネルの点滅も起きない
2. **UWP CoreWindow フォールバック**: 前面ウィンドウの直下に `Windows.UI.Core.CoreWindow` クラスの子ウィンドウがあれば（`FindWindowExW`）、`ImmGetDefaultIMEWnd` はその子に対して呼ぶ。パネルの表示位置・変化検知は従来どおりフレームウィンドウ基準
3. `--debug` の照会行に「（CoreWindow 経由）」を付記し、UWP 検知の成否を報告から判別できるようにする

## 理由

- タスクバー/デスクトップは入力モード表示が意味を持たない面であり、クラス名による判定は Windows のシェル実装として長期に安定している
- CoreWindow は UWP ホスティングの公開された構造で、フレームウィンドウ直下の子として取得できる。照会先を入力実体に近づける最小の変更で、失敗しても従来どおり「表示しない」に倒れる
- どちらもハードコードで足り、設定項目を増やさない（単純さの優先）

## 却下した代替案

- **`[ime_indicator]` に exe 除外リストを追加**: explorer.exe を除外するとエクスプローラーのファイル名変更・検索ボックスなど正当な用途まで消える。問題はプロセスではなくウィンドウの種類なので、クラス判定が正しい粒度
- **`GetGUIThreadInfo` / `AttachThreadInput` によるフォーカスウィンドウ取得**: 別プロセスのスレッド情報への依存や入力キュー結合は複雑でハングのリスクがあり、安定性優先の方針に反する
- **TSF（Text Services Framework）での UWP 状態取得**: 正攻法だが複雑度が跳ね上がる（ブリーフ §3.3 の Non-goals 理由そのもの）。CoreWindow フォールバックで不足する環境が確認されたら改めて判断
