# ADR 0008: ホットリロードは v0.1 ではトレイメニューのみとする

- ステータス: 承認
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

ブリーフ §3.1-5 は「設定のホットリロード（タスクトレイメニュー**または**ファイル監視）」を MVP 要件とする。ファイル監視を含めるかを決める。あわせて設定エラー時の挙動（開発計画 Phase 3 タスク 5）を確定する。

## 決定

1. **v0.1 のリロードはトレイメニューの「Reload config」のみ**とする。ファイル監視（保存で自動リロード）は v0.2 候補とし、必要になった時点で ADR を書く
2. **設定エラー時の挙動**（04_config-spec.md §4 に反映）:
   - 起動時にエラー → エラーを表示して**終了**（設定が効いていないのに常駐しない）
   - リロード時にエラー → **直前の有効な設定を維持**し、コンソールとトレイのツールチップにエラーを表示（設定編集ミスでリマップが止まらない）
3. リロードは新しい `RemapTable` を構築してから `ArcSwap::store` で一括差し替え（ADR 0003）。フックは常に一貫したテーブルを参照し、キーの取りこぼしは発生しない

## 理由

- ブリーフは「または」であり、トレイメニューだけで MVP 要件を満たす
- ファイル監視には notify 等の追加依存（ライセンス系統が MIT/Apache と異なる）と、エディタの多段書き込みに対するデバウンス処理が必要で、v0.1 の複雑さに見合わない（優先順位: 単純さ > 機能）
- 設定変更の頻度は低く、トレイから 1 クリックのコストは許容範囲

## 却下した代替案

- **notify クレートによるファイル監視**: 上記のとおり依存とデバウンスの複雑さが増える。将来の v0.2 で再検討
- **ポーリング（mtime 監視）**: 常駐ソフトとして無駄な定期 I/O を発生させる
- **リロードエラー時に空設定へフォールバック**: ユーザーの意図（現に動いていた設定）を勝手に無効化するため不採用
