# ADR 0017: マクロ出力は修飾キーを差分遷移で送出する

- ステータス: 承認
- 日付: 2026-07-18
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

`"C-t" = ["C-Right", "C-Left", "C-S-Right"]`（単語選択マクロ）が「動作する時としない時がある」と実機で報告された。デバッグログではマクロ送出自体は毎回行われており、失敗はアプリ側の解釈で起きている。

従来実装は要素ごとに「押下中の全修飾キーを解除 → 要素の修飾キーを押下 → キー → 解除」を繰り返すため、物理 Ctrl が押しっぱなしの C-t では **1 バッチ内で Ctrl の up/down が 3 往復**していた。修飾キー状態をメッセージ同期で読むアプリ（従来型 Win32）は正しく解釈するが、**非同期に修飾キー状態を参照するアプリ**（WinUI / UWP 系の新しいメモ帳、Electron 系等）は、この高速な状態の揺れをサンプリングのタイミング次第で拾い損ね、「Ctrl なしの Right」等と誤読する。これが間欠的な失敗の原因と推定した。

## 決定

マクロの各要素間で、**現在の修飾キー状態と次要素の必要状態の差分だけ**を送出する（`plan_transition`: 純粋関数として単体テスト対象）。

- 物理的に押下中の修飾キーが要素の要求を満たしていれば**一切触れない**。C-t の例では Ctrl に触れず `Right↓↑ Left↓↑ Shift↓ Right↓↑ Shift↑` のみが送出され、手入力と区別のつかないイベント列になる
- 不要になったクラスの解除が Alt/Win を含む場合は直前にマスクキーをタップ（ADR 0015 と同じ規則）
- 最後に物理押下状態への復元も差分で行う
- 全体を 1 回の `SendInput` バッチで送る点は従来どおり（実入力の割り込み防止）

## 理由

- 修飾キーのイベント数が最小になり、非同期読みのアプリが誤読する「揺れの窓」がそもそも発生しない。上記の例では修飾キーイベントは Shift の 1 往復だけになる
- 副次的にイベント総数も減る（C-t で 16 → 8 イベント）

## 却下した代替案

- **要素間に数 ms の Sleep を入れる**（Keyhac / AutoHotkey の keydelay 方式）: フックコールバック内のブロッキング禁止（AGENTS.md 不変条件 2）に違反し、マクロ中は全システムの入力を遅延させる。差分遷移で不十分だった場合の次善策として保留
- **メッセージループ側でのマクロ遅延実行**: コールバック外に出せば Sleep は許されるが、マクロ完了前に後続の物理打鍵がアプリへ届き得る（順序逆転）ため、高速タイピング時の正しさを損なう
