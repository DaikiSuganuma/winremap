# ADR 0020: IME 状態インジケーターの本体統合（Non-goals の一部緩和）

- ステータス: 承認（オーナー指示）
- 日付: 2026-07-19
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

ブリーフ §3.3 は「IME の制御・切り替え」を Non-goals としてきた（IMM32/TSF は複雑度が跳ね上がるため）。一方でオーナーから、IME がオンになった瞬間にアクティブウィンドウ中央へ半透明インジケーターを一時表示する機能（調査資料「IME状態連動型インジケーター」）を winremap に統合する指示があった。

今回の機能は IME の**状態表示のみ**であり、IME の制御・切り替えは行わない。ただし IMM32 API の照会という Non-goals が避けてきた領域に踏み込むため、スコープの再定義と設計上のガードレールを記録する。

## 決定

1. **IME 状態の「表示」のみを採用**する。IME の制御・切り替え（Emacs 日本語入力モード等）は引き続き Non-goal
2. 実装は **`src/ime_indicator/` ディレクトリに隔離した独立機能**とし、既存コードへの接点（main / hook / window）は通知 1 行レベルに限定する。詳細設計は [docs/08_ime-indicator-design.md](../05_ime-indicator-design.md)
3. 既定は **無効（opt-in）**。設定ファイル `[ime_indicator] enabled = true` で有効化する
4. 実装前に**技術検証フェーズ（Go/No-Go）を必須**とする。Windows 11 のモダン Microsoft IME で `IMC_GETOPENSTATUS` が実際に信頼できるかをオーナー環境で確認してから本実装に進む（[docs/09_ime-indicator-plan.md](../06_ime-indicator-plan.md) Phase I1）
5. IME 状態の照会・オーバーレイ描画は**専用スレッド**で行い、フックコールバック内では非ブロッキングな `PostThreadMessageW` による通知のみを許す（不変条件 2 の第 3 の明示的例外として AGENTS.md / ブリーフ §5 を改訂する。改訂はオーナーレビュー必須）

## 理由

- winremap は常駐基盤（メッセージループ、トレイ、単一インスタンス、TOML 設定＋ホットリロード、`SetWinEventHook` によるフォアグラウンド監視、WH_KEYBOARD_LL フック）を既に持ち、本機能が必要とするインフラの大半を再利用できる。特に「モダン IME で照会 API が機能しない場合のフォールバック＝トグルキーのグローバル監視」はフックの副産物として無償で手に入る
- 常駐プロセス・トレイアイコン・配布物を 1 つに保てる（単一バイナリ方針と一致）
- 表示のみであれば、Non-goals が避けたかった本丸（IME 制御による入力状態の破壊リスク）には踏み込まない
- 検証フェーズと opt-in 既定により、「安定性 > 単純さ > 機能」の優先順位を保ったまま導入できる

メリット・デメリットの全量評価は設計書 §2 に記載。

## 却下した代替案

- **別バイナリ（ワークスペース内の独立クレート）として作る**: winremap 本体の安定性を完全に守れるが、常駐プロセス・トレイ・設定・配布物が 2 つになり、トグルキー検知のためだけに 2 本目のキーボードフックを常駐させることになる（二重フックはブリーフ §9 が併用非推奨とする状況を自作することに等しい）。統合し、モジュール隔離と opt-in で安定性リスクを抑える方を選んだ
- **OS 標準機能（Microsoft IME の「入力モードの切り替えの通知」）に任せる**: 画面中央への「あ」/「A」表示は標準で存在するが、表示位置がアクティブウィンドウ中央ではなく、表示時間・見た目の調整もできない。オーナーの要件（ウィンドウ中央・カスタマイズ）を満たさない
- **TSF（Text Services Framework）による正式な状態取得**: 最も将来性があるが複雑度が跳ね上がり、Non-goals を設けた理由そのもの。まずレガシー IMM32 照会＋フック推測のハイブリッドで成立するかを検証する
- **Cargo feature でのコンパイル時分離**: CI のビルドマトリクスが増え、feature 組み合わせの保守が発生する。実行時 opt-in（既定無効）とディレクトリ隔離で「別機能」の分離目的は満たせるため不採用
- **フックコールバック内での IME 状態照会**: `SendMessage` 系は相手プロセスの応答を待ちブロックし得るため不変条件 2 に真っ向から違反する。検討の余地なく却下
