# ADR 0050: 設定ファイルの切り替えはパスの一本化＋既存リロード経路で行う

- ステータス: 承認（オーナー承認 2026-07-22、[画面設計書](../04_config-editing-screens.md) の承認に伴う）
- 日付: 2026-07-22
- 作成: Claude Code（AI モデル: claude-opus-4-8）

## 文脈

v0.4 の設定ウィンドウはエクスプローラー風のアドレスバーを持ち、設定フォルダー直下の `*.toml` をプルダウンで選んで**使用中の設定ファイルを切り替えられる**（[画面設計書 §2.3](../04_config-editing-screens.md)、オーナー指示 2026-07-22）。

現状、設定ファイルのパスは起動時に 1 回決まり（`-c` または既定 `%APPDATA%\winremap\config.toml`）、**2 箇所に複製される**:

- `gui::config_path()` — `Mutex<PathBuf>`。設定ウィンドウの表示・エディタ起動が読む（`gui::set_config_path` で起動時に設定）
- `tray::Tray::config_path` — `init` で渡されるコピー。**リロードの実体（`Tray::reload`）はこちらを読む**

GUI でパスを切り替えても、リロードがトレイのコピーを読む限り旧ファイルが読み込まれる。切り替え機能にはパスの一本化が要る。

## 決定

1. **プロセスの「使用中設定パス」の唯一の正を `gui::config_path()`（既存の `Mutex<PathBuf>`）とする。** `tray::Tray` は自前の `config_path` フィールドを廃し、リロードのたびに `gui::config_path()` を読む。
2. **切り替え＝「パスを差し替えて、既存のリロード経路に載せる」。** アドレスバーで別ファイルを選ぶと `gui::set_config_path(新パス)` → `gui::request_reload()`。以降はトレイの再読み込みと同一の経路（`main` のメッセージループ → `Tray::reload_now` → `config::load` → atomic swap、[ADR 0003](../../v0.1/decisions/0003-config-reload-arc-swap.md)）。切り替え専用の読み込み経路は作らない。
3. **切り替え先が読めない・検証エラーのときは、現行のリロード失敗と同じ扱い。** ライブ表は据え置き（リマップは止まらない）、エラーを通知し、パスは切り替わったまま（壊れたファイルへの `-c` 起動や、使用中ファイルを壊す編集と同じ意味論）。ステータスバーには失敗の旨を出す。
4. **切り替えは閲覧モードでのみ可能。** 編集モード中はプルダウンを無効化する（編集内容の行き先が失われるため。画面設計書 §2.4）。

## 理由

- **リロードの意味論を 1 つに保つ。** 読み込み・スワップ・IME/マクロ同期・失敗時のライブ表据え置きは `Tray::reload` に集約されている（v0.1 以来の実績）。切り替えを「パス差し替え＋同じ経路」と定義すれば、新しい読み込みコードを书かず、失敗時の挙動も既存と自動的に一致する。
- **一本化の向きは「トレイが GUI の正を読む」が最小。** `gui::config_path()` は既に `Mutex<PathBuf>` で可変・スレッド安全であり、書き手（起動時の `main`、アドレスバー）も読み手（設定ウィンドウ・エディタ起動・トレイのリロード）も全てフック外。フックコールバックはパスに触れないため、不変条件 2 に影響しない。
- **パスが切り替わったまま失敗を見せるのは、既存の失敗モデルと同型。** 現行でも使用中ファイルを不正に編集してリロードすれば「パスは有効・表は旧のまま・エラー通知」になる。切り替え失敗だけ自動で巻き戻すと、状態が「見えないところで元に戻る」ことになり、アドレスバーの表示と利用者の操作が食い違う。

## 却下した代替案

- **トレイのコピーを残し、切り替え時に GUI からトレイへ新パスを通知する**: 通知チャネルの追加とコピーの同期という新しい失敗様式（片方だけ更新される）を持ち込む。正を 1 つにすれば同期問題は存在しない。
- **パスの正を新モジュール（例 `config::active_path`）へ移す**: 置き場所としては中立だが、既存の読み手・書き手を全て書き換える churn に見合う利得がない。`gui` はフック外の共有 UI 状態の置き場として既に機能している。
- **切り替え失敗時にパスを旧ファイルへ自動で巻き戻す**: 一見親切だが、「選んだのに戻っている」という暗黙の状態変化を生む。失敗を見せて利用者に次の操作（直す・別ファイルを選ぶ・↻）を委ねる方が、通知した上で何もしない現行の失敗モデルと一貫する。
- **切り替え時にフルの検証をしてから切り替える（プリフライト）**: 検証が 2 回（プリフライトとリロード）になり、その間にファイルが変わる TOCTOU を抱える。リロード経路の検証 1 回で足りる。
