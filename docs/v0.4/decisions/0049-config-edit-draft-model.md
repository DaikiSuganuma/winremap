# ADR 0049: 設定 GUI 編集は「閲覧＝ライブ表 / 編集＝ファイル起点ドラフト」の二モデルで行う

- ステータス: 承認（オーナー承認 2026-07-22）
- 日付: 2026-07-22
- 作成: Claude Code（AI モデル: claude-opus-4-8）

## 文脈

v0.4 Phase A で設定 GUI の編集・保存（マイルストーン B2〜B4）を実装する（[v0.4 開発計画](../01_development-plan.md) §2、[設計書](../02_config-editing-design.md)）。

v0.2 で出荷済みの B1（閲覧）は、**フックが引くライブの `RemapTable`**（`hook::REMAP_TABLE`、`Arc<RemapTable>`）を毎フレーム読んで表示する作りになっている（`src/gui/config_window.rs`）。この `RemapTable` は次の性質を持つ。

- **コンパイル済み・検証済みの実行時形**である。`"S-C-h"` は `C-S-h` へ正準化され、`BTreeMap`／`HashMap` に格納されて**ソースの並び順は失われている**
- **コメントを持たない**。コメントは表示のためだけに `config::comments::read(path)` で別途ファイルから読み直している
- **フックの原子的スワップ（[ADR 0003](../../v0.1/decisions/0003-config-reload-arc-swap.md)）で、いつでも裏で差し替わりうる**。トレイのリロード、マクロ記憶の状態変化などが、GUI の都合と無関係にこれを入れ替える

v0.2 設計書 §4.1 は編集の中間表現 `ConfigDraft` を「`config.toml → ConfigDraft → 保存」という一方向で描いていたが、B1 が実際にはライブ表を表示する作りで出た結果、**「画面に出ているもの（ライブ表）」と「編集して保存するもの（ファイル）」が別物である**ことを、編集機能の設計として明示的に解く必要が生じた。

同時に守るべき制約:

- **手書き設定の保全**（[ADR 0036](../../v0.2/decisions/0036-config-save-preserves-comments.md)）。コメント・並び順を壊さない
- **未検証の入力途中を保持できること**（v0.2 §4.1）。`"Bak"` のような値を持てないと編集できない
- **編集中にライブ表が差し替わっても編集内容が消えないこと**

## 決定

1. **二つのモデルを分ける。**
   - **閲覧モデル** = ライブの `RemapTable`（現行 B1 のまま）。読み取り専用表示に使う
   - **編集モデル** = **ファイルを起点に構築する独立した `ConfigDraft`**（`src/config/draft.rs` を新設）。未検証の文字列を保持できる、`raw` に近い文字列ベースの型

2. **ドラフトはライブ表からではなく、ファイル（`toml::from_str`）から作る。** ［編集］押下時にファイルを 1 回読んでドラフト化する。ライブ表とファイルが食い違う瞬間（保存直後の未リロード等）でも、編集は常にファイルの実体から始まる。

3. **編集中、ドラフトはライブ表から独立して生きる。** 編集モード中に `REMAP_TABLE` が裏で差し替わっても（トレイのリロード等）、ドラフトには触れない。フックは影響を受けない（GUI スレッド上の話であり、コールバックには一切関与しない）。

4. **保存の瞬間だけファイルへ書く。** 保存直前にファイルを `toml_edit::DocumentMut` として読み直し、ドラフトの差分だけを適用（ADR 0036 の往復保持）→ `config::parse_str` で検証 → 一時ファイル＋`rename` で原子的に置換 → `gui::request_reload()`（トレイと同じ [ADR 0003](../../v0.1/decisions/0003-config-reload-arc-swap.md) の経路）。保存に成功したらドラフトを破棄し、表示はリロード後のライブ表へ戻る。

5. **入力キーの綴りは正準化しない。** ドラフトは書かれた綴り（`"S-C-h"`）を保持し、保存時もその綴りで書く。手書きの綴りを勝手に変えない。

## 理由

- **ライブ表は編集の起点にできない。** 正準化・並び替え・コメント喪失を経た実行時形へ書き戻すことは、手書き設定の保全（ADR 0036）と両立しない。編集の起点はソースの実体であるファイルしかない。
- **`ConfigDraft` を別に持つのは v0.2 で既に決めた方向**（v0.2 §4.1）であり、本 ADR はそれを「ライブ表とは別モデル」と明示して具体化しただけで、新しい構造を発明していない。
- **既存部品でそのまま組める。** `toml_edit` は依存済み（`config::comments` が読み取りで使用中）で保存に追加依存が要らない。保存前検証は既存の `config::parse_str`（→ 行・列つき `Issue`）を再利用でき、CLI・トレイと同一の検証器になる。前面アプリ取得は既存の安全な公開関数 `window::with_foreground_exe` で足り、GUI に `unsafe` を持ち込まない（不変条件 3）。保存後リロードは既存の `request_reload` に載る。
- **ライブ表とドラフトを独立させると、編集中のリロード・マクロ記憶と衝突しない。** 編集はファイルからの独立コピーなので、裏でライブ表が変わっても消える筋合いがない。逆に、ライブ表を直接編集する設計だと atomic swap との競合を GUI 側で調停する羽目になり、不変条件 2（フックを止めない）に対するリスクを GUI に持ち込む。
- **保存を 1 トランザクションに閉じられる。** 「読む→差分適用→検証→原子的置換→リロード」が一続きで、外部変更検出（mtime+size の一点比較）とも噛み合う。フックとフックのメッセージループには一切触れないため、安定性優先の原則に反しない。

## 却下した代替案

- **ライブ `RemapTable` を直接編集して書き戻す**: 画面に出ているものをそのまま編集できて一見素直だが、`RemapTable` は正準化・並び替え・コメント喪失を経た実行時形で、ここから元のファイルを復元できない。ADR 0036 の手書き保全に正面から反する。さらに atomic swap（ADR 0003）が編集中に走ると、GUI が保持する参照と実体がずれ、その調停を GUI 側に持ち込むことになる。

- **編集中はライブ表のスワップを止める（ロックする）**: ドラフトとライブ表の乖離を避けられるが、フックのリロード経路に GUI 都合のロックを差し込むことになり、不変条件 2（フックコールバック内でロック待ちをしない、フックを止めない）の精神に反する。編集は数分に及びうるため、その間リロードを封じるのは運用上も不便。

- **ドラフトをライブ表から作る（ファイルを読み直さない）**: ファイル I/O を 1 回省けるが、ライブ表はコメントも並び順も綴りも失っているため、そこからドラフトを作ると往復保持（ADR 0036）が最初から成立しない。読み直しのコストは編集開始時の 1 回のみで、割に合う。

- **`RawConfig` を編集用に流用する**: 既存のデシリアライズ型を使えば型を 1 つ増やさずに済むが、`RawConfig` は `serde` 専用で `Spanned`／`deny_unknown_fields` を持ち、未検証の入力途中（`"Bak"`）を自然には保持できない。編集用途に噛み合わないため、文字列ベースの `ConfigDraft` を別に持つ。

- **保存のたびに全体を書き直す（`DocumentMut` 差分ではなくドラフトから TOML を再生成）**: 実装は単純になるが、コメント・空行・並び順・綴りといった「地」を全て失う。ADR 0036 が防ごうとしたものそのものであり、却下。
