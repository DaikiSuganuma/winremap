# ADR 0051: 設定ファイルの変更検知に notify クレートを使う（変更ありマーク）

- ステータス: 承認（オーナー承認 2026-07-22、[画面設計書](../04_config-editing-screens.md) の承認に伴う）
- 日付: 2026-07-22
- 作成: Claude Code（AI モデル: claude-opus-4-8）
- 参照: [notify クレート](https://docs.rs/notify/)、[`ReadDirectoryChangesW`（Microsoft Learn）](https://learn.microsoft.com/en-us/windows/win32/api/winbase/nf-winbase-readdirectorychangesw)

## 文脈

v0.4 の設定ウィンドウは、使用中ファイルの中身が読み込み済み状態と食い違ったらアドレスバーに変更ありマーク（●）を出す（[画面設計書 §2.2](../04_config-editing-screens.md)）。

現行 B1 は表示用の更新日時を **2 秒間隔のポーリング**（`ConfigWindow::file_time`、`(mtime)` の stat）で読んでいた。オーナーから「ポーリングではなく notify クレートを使えないか」の指示（2026-07-22）。

`notify` は Rust の標準的なファイル監視クレートで、Windows 実装は OS のディレクトリ変更通知 `ReadDirectoryChangesW` を使う。監視スレッドをクレート側が持ち、イベントをコールバックで届ける。

## 決定

1. **`notify` クレートを依存に追加し、設定フォルダー（使用中ファイルの親、非再帰）を監視する。** ポーリングは平常経路から外す。
2. **監視は設定ウィンドウが表示されている間だけ。** 表示で `watch`、非表示（閉じる＝隠す、[ADR 0032](../../v0.2/decisions/0032-log-window-hide-instead-of-close.md)）で `unwatch`。ウィンドウを開いていない間は監視スレッドも遊ばせない。
3. **イベントは「何かが変わった」の合図としてだけ使う。** イベント受信でフラグを立てて `egui::Context::request_repaint()`（スレッド安全）を呼び、GUI スレッドが次フレームで `(mtime, size)` を 1 回比較して ● を確定する。イベントの種別・パスの解釈に依存しない（エディタの保存は一時ファイル＋rename・複数イベントで届くため）。デバウンス用の追加クレート（`notify-debouncer-*`）は入れない。
4. **監視の開始に失敗したら、現行の 2 秒ポーリングへフォールバックする。** マークの反応が遅れるだけで、機能は失わない。
5. **自動リロードはしない。** ● を付けるだけで、読み込むのは利用者が ↻ か切り替えを押したとき。ファイル監視による自動リロードを見送った v0.2 の判断（v0.2 開発計画のスコープ外項目）は変えない。
6. **監視スレッドはフック・フックのメッセージループに一切触れない**（不変条件 2）。触れるのはフラグ（atomic）と `request_repaint` のみ。`unsafe` も増えない（不変条件 3。`notify` の API は安全）。

## 理由

- **イベント駆動は「開いている窓に外部変更が映るまでの遅さ」を消す**。ポーリングの 2 秒は表示専用なら許容だったが、● は「保存したのに読み込まれていない」を伝える状態表示であり、即時性に意味がある。
- **合図＋stat 確定の分業が最も安定する。** `ReadDirectoryChangesW` 由来のイベントは保存 1 回に対して複数届き、エディタによっては rename・作成・削除の組み合わせになる。イベントを真に受けて状態遷移すると解釈が OS・エディタ依存になる。「イベント→ stat 1 回」なら判定は常に単純な事実比較で、デバウンスも不要。
- **依存の追加に見合う。** 自前で `ReadDirectoryChangesW` を叩くと `unsafe`・ハンドル管理・スレッドを自前で持つことになり、不変条件 3（unsafe の隔離）に新しい島を増やす。`notify` は成熟した定番（cargo エコシステムでのデファクト）で、安全 API の裏に隔離してくれる。
- **ライセンスは許諾的で MIT と両立する**（CC0-1.0 系。導入時に `cargo metadata` で確定値を確認し、本 ADR に追記する）。コードの移植ではなく通常の依存利用なので THIRD-PARTY-NOTICES.md の対象外（AGENTS.md の規定はコード移植に対するもの）。

## 却下した代替案

- **現行の 2 秒ポーリングを続ける**: 実装は最小だが、●の即時性が落ち、ウィンドウ表示中は永久に stat が走る。オーナー指示とも合わない。フォールバックとしてのみ残す。
- **`ReadDirectoryChangesW` を自前実装**: 依存は増えないが `unsafe`・スレッド・ハンドル寿命を自前管理することになり、安定性優先の原則に反する。監視は本業ではない。
- **`notify-debouncer-mini` / `notify-debouncer-full` も導入**: イベント列を平滑化してくれるが、本件の消費側は「フラグを立てて stat」だけなので平滑化する対象がない。依存は少ないほどよい。
- **イベントの種別・パスを解釈して ● を直接更新**: stat 1 回を省けるが、判定が OS・エディタの保存様式に依存する。事実（今のファイルの `(mtime, size)`）で判定する方が単純で正しい。
- **常時監視（ウィンドウ非表示中も）**: 切り替え・編集はウィンドウからしか起きないので、非表示中の監視は誰も見ない情報を集めるだけ。表示中に限れば資源も挙動も説明しやすい。
