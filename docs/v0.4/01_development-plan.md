## 3. Phase B — ログ表示への入力文字コード・制御文字コード表示

### 3.1 背景と狙い（オーナー決定: 2026-07-22）

本プロジェクトの発端は **Ctrl+H（`0x08` BS）と Backspace（端末上で `0x7f` DEL）の食い違い**だった（ブリーフ §1.1）。現在の「ログを表示」ウィンドウ（[ADR 0029](../v0.3/decisions/0029-attach-console-and-tray-log-window.md)、`src/gui/log`）は、キーイベントを**キー名レベル**（`C-h` 等）で表示するが、そのキーが実際に生む**文字コード・制御コード**は出さない。これを添えると、まさに発端となった種類の「見た目は同じでも端末に届くコードが違う」問題を、利用者が設定ファイルやドキュメントを開かずに診断できる。

### 3.2 不変条件 6（キーロガー化の禁止）との整合 — **ADR で境界を固定する**

これは**セキュリティ上最も注意を要するフェーズ**である。AGENTS.md 不変条件 6・ブリーフ §5-6 は「キー入力内容のログ保存機能を実装しない。デバッグログはキー名レベルまで、既定 OFF」と定める。文字コードの表示がこの境界のどちら側かを、実装前に ADR で明確に決める。

論点として少なくとも次を ADR で解く。

- **既定 OFF・ディスク非保存の維持**: デバッグログは今も既定 OFF で、ウィンドウを開いている間だけ有効・ディスクには一切書かない（[ADR 0029](../v0.3/decisions/0029-attach-console-and-tray-log-window.md)）。文字コード表示もこの制約の内側に完全に収める。**表示する情報の粒度が上がるだけで、保存経路は増やさない**
- **「キー名レベル」の解釈**: 制御コード（Ctrl+H → `0x08`、Enter → `0x0d` 等）は物理キー／VK から決まる**そのキーの性質**であり、既存のキー名表示と同粒度と見なせるか。一方、印字可能文字（`a` → `U+0061` 等）の逐一表示は「打った文字列」の記録に近づく。どこまでを出すか（制御コードのみか、印字可能文字も含むか）を決める
- **算出方法**: `MapVirtualKey` / `ToUnicode` 等で VK・scan から求めるのか、リマップ後の出力に対して求めるのか（マクロ記憶が「リマップ後の出力」を対象にした前例＝[ADR 0043](../v0.3/decisions/0043-runtime-macro-recording.md)と整合させる）。フックコールバック内で重い Win32 API を呼ばない制約（不変条件 2）に触れないよう、算出は `drain_debug_log`（メッセージループ上）で行う
- **表示形式**: `U+0008 (BS)` のような Unicode コードポイント＋制御文字略号か、端末が送るバイト列（`0x7f`）か、両方か。発端の問題（BS と DEL の違い）が読み取れる形にする

### 3.3 タスク

1. ADR（次の空き番号。0049 は Phase A の [ADR 0049](decisions/0049-config-edit-draft-model.md) が使用済み）— ログへの文字コード・制御コード表示の採否と、不変条件 6 の境界（何を出し何を出さないか、既定 OFF とディスク非保存の維持、算出の場所と方法）
2. 実装 — `drain_debug_log`（`src/hook.rs`）で各イベントに文字コード情報を付す。フックコールバック（`log_debug`）には手を入れない（不変条件 2 の遵守）。UI 文字列は `src/i18n.rs` を通す（§5）
3. テスト — VK／scan → 文字コード・制御コードの導出を**純粋関数として単体テスト**（境界の代表: Ctrl+H=`0x08`、Backspace、Enter、印字可能キー、非文字キー）
4. 受け入れチェックリスト（`docs/v0.4/03_acceptance-checklist.md`）へ表示確認の項目を追加

### 3.4 完了条件

- 「ログを表示」ウィンドウで、キーイベント行にそのキーの文字コード・制御コードが読み取れる
- 発端の問題（Ctrl+H の `0x08` と Backspace の `0x7f`）の違いがログ上で判別できる
- デバッグログは引き続き既定 OFF・ディスク非保存で、表示の粒度が上がっても保存経路は増えていない（不変条件 6 の遵守）
- 表示中もリマップが停止・遅延しない（不変条件 2 の遵守）

## 5. 多言語化を見据えた構造化（v0.4 では移行しない）

多言語ファイル化そのものは見送るが（§1）、v0.4 は Phase A・B で**新しい UI 文字列を増やす**。v0.3 §5 と同じ規律を守り、将来 Project Fluent / ICU4X へ移す際の作業が「文字列を写すだけ」で済むようにする。

- **UI 文字列は必ず `src/i18n.rs` を通す。** 他のモジュールに直書きしない
- **キーと値を分けた形を保つ。** `Texts` のフィールド名がそのまま翻訳キーになる想定で命名する。文字列連結で文を組み立てず、書式つきメッセージは引数を取る関数として 1 箇所にまとめる
- **語順に依存した組み立てをしない。** 文全体を 1 つのメッセージとして持つ
- **技術診断メッセージは英語のまま。** 設定検証エラー・`anyhow` のコンテキストは「issue にそのまま貼れる」ことを優先する現行方針を維持する（Phase B の文字コード表記＝`U+0008 (BS)` 等の識別子部分も英語表記のまま）

移行時に答えを出す論点（バイナリサイズ、埋め込みか実行時読込か、欠損キーのフォールバック、`--lang` の BCP 47 拡張）は v0.3 §5 に記録済み。参照 URL もそちらにある。
