## 0. 本書の位置づけ（v0.2 設計書との関係）

v0.2 設計書はキーマップ編集まで含む全体（B0〜B5）を設計済みで、**画面構成（§3）・編集モデルの意図（§4）・エラー表示（§5）・不変条件（§6）は今も有効**である。ただし v0.2 設計書が書かれた時点では B1（閲覧）が未実装で、実際に出荷された B1 の作りが設計の一部を具体化・変更した。本書はその差分を埋め、B2〜B4 を**現行コードのどの関数・型に接続するか**まで確定させる。B5（README／サイト反映）は v0.4 開発計画の Phase C で扱う。

**v0.2 設計書から更新する点**（詳細は §1）:

- v0.2 §4.1 は「`config.toml → ConfigDraft → 保存」の一方向を描くが、B1 は**ライブの `RemapTable`（フックが引く実体）を表示する**作りで出荷された。編集はこのライブ表を書き換えられない。**閲覧＝ライブ表、編集＝ファイル起点のドラフト**という二モデルを明示する（ADR 0049）
- `toml_edit` は既に依存に入っており（`src/config/comments.rs` が使用）、保存は**新規依存なし**でこの `DocumentMut` 経路を拡張する
- 保存前検証は既存の `config::parse_str`（→ `Vec<Issue>`、行・列つき）をそのまま再利用する。v0.1 の「全件まとめて行番号つき報告」がこれで担保される

## 2. 画面構成（B1 からの差分）

## 3. 編集モデル（ADR 0049）

### 3.1 二つのモデルとモード

```
閲覧モード:  hook::REMAP_TABLE (Arc<RemapTable>) ──表示──▶ 画面（読み取り専用）
                    ＋ config::comments::read(path)（コメント）

［編集］押下 = ファイルを 1 回読む
              config.toml ──toml::from_str──▶ ConfigDraft（未検証の文字列を保持）
                        └──開いた時点の (mtime, size) を控える（§6 外部変更検出）

編集モード:  ConfigDraft ──表示・編集──▶ ConfigDraft'
              ［保存］= §4（DocumentMut へ差分適用 → 検証 → 原子的保存 → request_reload）
              ［元に戻す］= ドラフト破棄 → 閲覧モードへ
```

**なぜ二モデルか（ADR 0049 の核）**: B1 が表示するライブ `RemapTable` は、フックが引くために**コンパイル・検証・正準化された**実行時形で、ソースの並び順もコメントも失われている。ここへ直接書き戻すことはできず、手書き設定の保全（ADR 0036）とも両立しない。一方、編集中にトレイ操作やマクロ記憶がライブ表を差し替えても（`REMAP_TABLE` の atomic swap は編集と無関係に起きる）、編集内容が消えてはならない。ゆえに**編集はライブ表から切り離したファイル起点のドラフト上で行い、保存の瞬間だけファイルへ書く**。

### 3.2 `ConfigDraft` の表現

**未検証の文字列を保持できること**が要件（`"Bak"` のような入力途中を持てないと編集できない。v0.2 §4.1）。したがって検証済みの `RemapTable` ではなく、**`raw` に近い文字列ベースの独立した型**を新設する（`src/config/draft.rs`）。`RawConfig` は `serde` 専用で `Spanned`／`deny_unknown_fields` を持ち編集に不向きなため流用しない。

```
ConfigDraft {
    keymaps: Vec<KeymapDraft>,          // 画面の左一覧＝ファイル順
    macro_delay: String,                // 全体設定。空 = 未設定
    macro_record: MacroRecordDraft,     // start/stop/play の文字列
    ime_indicator: ImeIndicatorDraft,   // enabled/duration/size/opacity/show_app_name/triggers
    // v0.1 綴り macro_delay_ms を使っているファイルはその綴りを保持する
    uses_legacy_delay_key: bool,
}
KeymapDraft {
    name: String,
    application: Vec<String>,           // exe 名 or "*"
    exclude: Vec<String>,               // application=="*" のときのみ意味を持つ
    rules: Vec<RuleDraft>,              // ファイル順を保持（正準化しない）
}
RuleDraft { input: String, output: String }  // 入力・出力ともキー記法の生文字列。output はカンマ区切りでマクロ
```

- ドラフトは `ConfigWindow` 構造体に `Option<ConfigDraft>` として持つ（`Some` = 編集中）。フレームをまたいで生存する既存の置き場所（`Arc<Mutex<ConfigWindow>>`）にそのまま乗る
- ドラフトの構築元は**ライブ表ではなくファイル**（`toml::from_str`）。ライブ表とファイルが食い違う瞬間（保存直後の未リロード等）でも、編集は常にファイルの実体から始まる
- **正準化しない**。`"S-C-h"` と書いたら `"S-C-h"` のまま保持し、保存時もその綴りで書く（手書きの綴りを勝手に変えない）

### 3.3 ライブ表示との整合

- 編集モード中、画面はドラフトを表示する。ライブ表は裏で差し替わってもよい（フックは影響を受けない）
- 「他のキーマップ」列（B1 の `shared_inputs`）は編集中も出す。ドラフト内で計算する（保存前でも重複を気づけるように）
- 記憶マクロ表示（`recorded_macro_ui`）はライブの実行時状態なので、編集中もライブから読む（ドラフトの対象外）

### 3.4 ドラフトの寿命

| 契機 | 挙動 |
|---|---|
| ［編集］押下 | ファイルを読み `Some(draft)` を作る。読めなければ編集に入らず、フッターに理由 |
| ［保存］成功 | ファイルへ書き `request_reload()`。ドラフトは破棄し閲覧へ（表示はリロード後のライブ表へ戻る） |
| ［元に戻す］／編集中に窓を閉じる | ドラフト破棄。閉じるは v0.2 の「隠す」（ADR 0032）だが、**未保存の編集がある状態で閉じたら確認する**（§9 未決 4） |
| 編集中のトレイ「再読み込み」 | ライブ表は変わるが**ドラフトは触らない**。編集はファイルからの独立コピーであり、リロードで消える筋合いがない |

## 4. 保存（B2）

### 4.1 手順

```
1. 保存直前に (mtime, size) を再取得し、開いた時点と比較（§6 外部変更検出）
2. その時点のファイルを toml_edit::DocumentMut として読む（コメント・並び順の“地”）
3. ドラフトの差分だけ DocumentMut に適用（§4.2）
4. DocumentMut を文字列化し、config::parse_str で検証（§4.3）
   - 失敗: 保存しない。Issue をフッター＋該当欄に出す
5. 検証成功: 一時ファイルへ書き、rename で置換（§4.4）
6. gui::request_reload()（トレイと同じ経路。ADR 0003 の atomic swap）
```

「地」を**保存直前のファイルから**読むのは、開いてから保存までに手書きで足されたコメント等を保存が消さないため。外部変更があれば手順 1 で止まる（§6）。

### 4.2 差分適用（ADR 0036 の往復保持）

v0.2 §4.2 の規則を `DocumentMut` 操作へ具体化する。`comments.rs` が既に `DocumentMut` の読み取りで通っている構造（`doc["keymap"]` は `ArrayOfTables`、`[keymap.remap]` は `Table`）をそのまま書き込みに使う。

| 操作 | DocumentMut 上の扱い |
|---|---|
| **値の変更** | 該当キーの値のみ差し替え。行末コメント（`key = "v"  # メモ`）と装飾（`decor`）は保持される（`toml_edit` の既定挙動） |
| **項目の追加** | 対応するテーブル／配列の**末尾**に挿入。既存項目の間に割り込まない |
| **項目の削除** | 該当キーを消し、**直前に密着した（空行を挟まない）コメント行**も一緒に消す。空行で離れたコメントは残す |
| **並べ替え** | `[[keymap]]` テーブルごと、密着コメントを含めて移動 |
| **正準化の非適用** | 入力キーは書かれた綴りで書き戻す（§3.2）。ドラフトが `"S-C-h"` を保持していればそのキーで書く |

判断基準は v0.2 §4.2 のとおり「**保存後の差分が利用者にとって説明可能であること**」。4 パターン（変更・追加・削除・並べ替え）を往復テストで固定する（§8）。

### 4.3 保存前検証（既存資産の再利用）

- 文字列化した TOML を `config::parse_str` に渡す。返る `Vec<Issue>`（行・列・メッセージ）を検証エラーとして使う
- これは v0.1 CLI・トレイと**同一の検証器**であり、GUI 独自ルールを二重持ちしない（設定仕様 §4 の「全件まとめて報告」を GUI でも自動で満たす）
- `Issue.line/column` は文字列化後の TOML に対する位置。ドラフトの各欄への逆写像は、ドラフト構築時に「欄 → DocumentMut 上のパス」を持っておき、パス → 位置で対応づける（§9 未決 1）

### 4.4 原子的保存

- 同ディレクトリに一時ファイル（`config.toml.tmp` 等）を書き、`fs::rename` で置換。書き込み途中のクラッシュで `config.toml` が壊れない（同一ボリューム内 rename の原子性に依存）
- 失敗（権限・ディスク）はダイアログではなくフッターに出す（GUI が前面にあるのだからモーダルにしない。v0.2 §5）
- **ディスクに書くのは `config.toml`（と一時ファイル）だけ**。ログは書かない（不変条件 6）

## 5. 検証と入力補助（B3）

### 5.1 キー記法の逐次検証

- 入力・出力欄の直下に、`keymap::parse_key_combo`／`parse_input_pattern` の結果を随時出す
  - 妥当: 人間語で解釈を表示（`C-h` → 「Ctrl + h」、`A-x h` → 「Alt+x に続けて h（2 ストローク）」）
  - 不当: 理由と候補（`Bak` → 「未知のキー名。`Back` のことですか？」）。候補は既知キー名との近さ（編集距離）で 1 つ出す
- 出力欄はカンマ区切りをマクロ（`["C-Right", "C-Left"]`）として解釈
- **キーを実際に押して取り込む「キーキャプチャ」は作らない**（v0.2 §3.3。フックとの調停が不変条件 1 に対して割に合わない）

人間語レンダリングとキー名候補は**純粋関数**として `keymap` 側に置き単体テストする（§8）。i18n の対象（§7）。

### 5.2 使えるキー名一覧

欄の横の `?` から、使用可能なキー名の一覧を出す。一覧は `keymap` が持つキー名テーブルから生成する（表示専用で新規データを二重に持たない）。

## 6. 「今の前面アプリから取得」・外部変更・リロード（B4）

### 6.1 前面アプリ取得（§3.4 v0.2）

- 対象アプリのチップ列の脇にボタン。押すと **3 秒カウントダウン**後、`window::with_foreground_exe(|exe| exe.to_owned())` で前面 exe 名を取り、`application` に足す
- `with_foreground_exe` は前面変化イベントでキャッシュ済みの名前を渡す**安全な公開関数**。GUI から `unsafe`／Win32 を直接呼ばない（不変条件 3 の隔離維持）
- カウントダウン中に対象アプリへ切り替えてもらう。設定窓が前面のまま押すと設定窓自身の exe を拾うため、カウントダウンはその回避策でもある

### 6.2 外部変更の検出（v0.2 §4.3）

- ［編集］でファイルを読んだ時点の **`(mtime, size)`** を控える
- ［保存］直前に再取得して比較。変わっていたら「ファイルが外部で変更されています。上書きしますか／読み直しますか」を問う。**黙って上書きしない**
- ファイル監視による自動リロードは対象外（v0.2 開発計画で見送り済み）。あくまで保存時の一点比較

### 6.3 保存後リロード（v0.2 §4.4）

- 保存成功で `gui::request_reload()`。メッセージループが拾い、`config::load(path)` → atomic swap（ADR 0003）→ `mark_config_loaded()`
- GUI 専用のリロード経路は作らない。結果はログ窓にも 1 行残る（既存経路）
- リロード後、窓は閲覧モードのライブ表表示へ戻る（§3.4）

## 7. スレッド構成・不変条件・多言語

v0.2 §6 のまま。差分と念押しのみ:

- 編集・保存・検証はすべて **GUI スレッド**上（ADR 0035）。フックコールバックには触れない
- 設定差し替えは既存の `REMAP_TABLE` atomic swap のみ（ADR 0003）。GUI から直接スワップしない（`request_reload` 経由）
- 前面アプリ名は `window::with_foreground_exe` 経由。GUI に `unsafe` を持ち込まない（不変条件 3）
- ディスクに書くのは `config.toml`。ログは書かない（不変条件 6）
- **新規 UI 文字列は `src/i18n.rs` を通す**（v0.4 開発計画 §5）。人間語の記法解釈・検証メッセージ・確認ダイアログ文言・ボタン名すべて。技術診断（`Issue` のメッセージ本文）は現行どおり英語のまま

## 8. テスト方針

v0.2 §8 を踏襲。**編集モデルは純粋ロジックとして単体テスト**、GUI 描画は手動確認。

自動テスト（`src/config` の変更にテストを伴わせる規則）:

- `ConfigDraft` ↔ ファイル文字列の変換（読み込み → ドラフト → 文字列化の同値性）
- `toml_edit` 差分適用の 4 パターン（**値変更・追加・削除・並べ替え**）を、コメント付き入力と期待出力文字列で固定
- 保存前検証が `parse_str` の `Issue` を正しく拾うこと（壊れた記法・範囲外の数値・記憶キー衝突）
- キー記法の人間語レンダリングとキー名候補（`Bak`→`Back` 等）

手動確認（受け入れチェックリスト `docs/v0.4/03_acceptance-checklist.md` に Phase A 項目として追加）:

- コメント・並び順の保持（往復編集）
- 保存 → 即反映（リマップ挙動が変わる）
- 壊れた入力が保存されない
- 外部変更の検出（編集中にエディタで書き換え → 保存時に確認が出る）
- 「今の前面アプリから取得」が対象アプリを拾う
- 編集中もリマップが停止・遅延しない（不変条件 1・2。A-12 と同じ位置づけ）
- 日本語入力（B-1〜B-8 の再確認）

## 9. 未決事項（実装時にオーナー判断）

v0.2 §9 の 3 点（トレイ項目名、初回自動起動、ウィンドウサイズ記憶）は据え置き。本書で新たに挙げる:

| # | 事項 | 論点 |
|---|---|---|
| 1 | 検証エラーの欄への逆写像の粒度 | `parse_str` は文字列化後 TOML の行・列を返す。これをドラフトの欄へ厳密に戻すか（欄 → パス → 位置の対応表を持つ）、フッターにメッセージだけ出して欄強調は近似で済ませるか。前者は正確だが実装量が増える |
| 2 | 編集の粒度 | v0.4 でキーマップの追加・削除・並べ替えまで入れるか、まず既存キーマップ内の規則・アプリ・全体設定の編集に絞り、キーマップ自体の増減は次に送るか（B2 の完了条件はコメント保持の往復。編集範囲は段階化しうる） |
| 3 | ［編集］に入る単位 | 窓全体で 1 つの編集モードにするか、セクション単位でインライン編集にするか。前者（窓モード）を推奨（保存が 1 トランザクションで、外部変更検出とも噛み合う） |
| 4 | 未保存で閉じる／リロードしたときの扱い | 閉じる（＝隠す、ADR 0032）ときに未保存編集があれば確認するか、黙って破棄するか。確認を推奨 |

推奨（#2・#3）: **窓全体で 1 つの編集モード**とし、v0.4 は**キーマップ内の規則・アプリ・全体設定の編集＋キーマップの追加/削除/並べ替え**まで入れる（v0.2 で「キーマップ編集まで含む全体」を機能範囲と決めているため）。ただし実装が重ければ #2 を段階化の逃げ道として残す。
