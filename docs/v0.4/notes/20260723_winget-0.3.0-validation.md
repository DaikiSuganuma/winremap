# winget 初回提出（v0.3.0）の検証失敗と、v0.4.0 での再提出（作業ノート）

- 作成日: 2026-07-23
- 作成: Claude Code（AI モデル: claude-opus-4-8）／判断: オーナー
- 対象: [microsoft/winget-pkgs PR #405731](https://github.com/microsoft/winget-pkgs/pull/405731)（`DaikiSuganuma.WinRemap` 0.3.0 の新規登録）
- 関連: [ADR 0052（CRT 静的リンク）](decisions/0052-static-crt.md)、[ADR 0045（パッケージ配布チャネル）](../v0.3/decisions/0045-package-manager-channels.md)、[03_release-operations.md §3](../../03_release-operations.md)

本ノートは「起きたこと・原因・オーナー決定」を後から追えるようにする記録。設計判断そのものは ADR 0052 が正。

## 起きたこと

winget への初回提出（v0.3.0、2026-07-22）で、マニフェスト検証は通ったが**実行ファイルの起動検証が失敗**した。モデレーターの自動検証コメント（2026-07-23 01:17 UTC）:

> `Executable C:\Users\validator\AppData\Local\Programs\WinRemap\winremap.exe returned exit code: -1073741515`
> `C0000135 / STATUS_DLL_NOT_FOUND` — Possibly due to a missing dependency（最も多いのは `Microsoft.VCRedist.2015+.x64`）

## 原因

Rust の MSVC ターゲットは既定で C ランタイムを動的リンクするため、`winremap.exe` が **`vcruntime140.dll` をインポート**していた。この DLL は VC++ 再頒布可能パッケージの一部で、クリーンな Windows には存在しない。開発機には必ず入っているため、リリースまで顕在化しなかった。

ローカルの `target\release\winremap.exe` に含まれる DLL 名文字列を調べて確認した（UCRT の `api-ms-win-crt-*` は Windows 10 以降 OS 同梱のため問題なし。非 OS の依存は `vcruntime140.dll` のみだった）。

## 実施した対応

1. **暫定（PR ブランチ）**: `installer.yaml` に `Dependencies.PackageDependencies: Microsoft.VCRedist.2015+.x64` を追加してプッシュ（2026-07-23 01:35 UTC、フォークの `DaikiSuganuma.WinRemap-0.3.0` ブランチ）
2. **根治（develop）**: `.cargo/config.toml` で `-C target-feature=+crt-static`（ADR 0052）。ビルド後に `vcruntime140.dll` と `api-ms-win-crt-*` のインポートが**すべて消えた**ことを確認済み

## 検証パイプラインの読み方（次回のために）

ラベルとコメントだけを見ると誤読する。ビルド番号と時刻を突き合わせること。今回の例:

| 時刻(UTC) | 出来事 |
|---|---|
| 01:17〜01:18 | `@wingetbot run` ×2 → build 372313 / 372314（**修正前**のマニフェスト） |
| 01:35 | 依存追加コミット |
| 01:36 | build 372318 起動（**修正後**） |
| 02:17 | 372313 が `failed` |
| 02:25 | `Validation-Executable-Error` ラベル再付与 ← **372313 の結果**。修正後の結果ではない |
| 02:38 | 372318 が `partiallySucceeded`（唯一の警告は Microsoft 内部ツール Guardian のルールセット取得失敗で、本件と無関係）で終了 |

ラベルが既に付いている場合、同じ結果なら再付与のイベントが出ないため、**ラベルの有無だけでは合否を判定できない**。Azure DevOps の公開 API でビルド単体の結果とタイムラインを確認するのが確実:

```
https://dev.azure.com/shine-oss/8b78618a-7973-49d8-9174-4360829d979b/_apis/build/builds/<buildId>?api-version=7.0
https://dev.azure.com/shine-oss/8b78618a-7973-49d8-9174-4360829d979b/_apis/build/builds/<buildId>/Timeline?api-version=7.0
```

## オーナー決定（2026-07-23）

**v0.3.0 の PR は取り下げ、winget への登録は v0.4.0 でやり直す。**

理由:

- リリース済み v0.3.0 のバイナリは差し替えられない（`InstallerSha256` が変わるため、公開済み資産の置き換えは配布物の同一性を壊す）。0.3.0 のままでは依存宣言に頼るしかない
- winget クライアントの `PackageDependencies` は**実験的機能**扱いで、設定で有効にしないと処理されない・複数依存だと最初の 1 つしか入らない等の既知の不具合がある（[winget-cli #2202](https://github.com/microsoft/winget-cli/issues/2202)、[#4679](https://github.com/microsoft/winget-cli/issues/4679)）。検証を通せたとしても、**実利用者の環境で VCRedist が入る保証がない**
- v0.4.0 のバイナリは ADR 0052 で静的リンク済みになり、**依存宣言そのものが不要**になる。GitHub Releases から直接落とす利用者も同時に救われる
- v0.4.0 のリリースが近く、0.3.0 を通すためだけに v0.3.1 を出すコストに見合わない

## v0.4.0 提出時にやること

1. PR #405731 を**クローズする**（v0.4.0 の提出が用意できてから。理由を添える）
2. `packaging/winget/*.yaml` を 0.4.0 に更新する。**`Dependencies` は入れない**（静的リンク済みのため）
3. 提出前に、リリースする exe に非 OS の DLL 依存が無いことを確認する（[03_release-operations.md §3](../../03_release-operations.md) の手順に追加済み）
4. `manifests/d/DaikiSuganuma/WinRemap/0.4.0/` として新規 PR を出す
