# ADR 0037: eframe のルートビューポートは不可視のホストにし、実ウィンドウは全て子にする

- ステータス: accepted（[ADR 0035](0035-gui-single-event-loop-multi-viewport.md) のウィンドウ構成を差し替える）
- 日付: 2026-07-21
- 作成: Claude Code（AI モデル: claude-opus-4-8）／レビュー・承認: オーナー

## 文脈

ADR 0035 では「ルートビューポート＝設定ウィンドウ、子ビューポート＝ログウィンドウ」とし、
起動直後は両方とも非表示にする設計にした。マイルストーン B0 で実装したところ、オーナーの
実機確認で次の不具合が出た。

> 最初にログ表示を行うと、設定ウィンドウも開いてしまう。一度閉じれば再度ログ表示しても
> 設定ウィンドウは開かない。

原因は eframe の仕様である。eframe は起動時の白いちらつきを避けるため、**最初のフレームを
描き終えた時点でルートウィンドウを自分で `set_visible(true)` する**
（`EpiIntegration::post_rendering`、egui PR #2279）。こちらが
`ViewportBuilder::with_visible(false)` で作っても、その後から上書きされる。「一度閉じれば
以後は開かない」のは、閉じたときにこちらが送る `Visible(false)` が残るためで、症状と一致する。

次のフレームで `Visible(false)` を送り直す対症療法を入れたが、オーナーの再確認で
「設定ウィンドウが一瞬開く」ことが分かった。1 フレーム分は必ず表示されるため、この方向では
ちらつきを消せない。

参照:

- egui PR #2279（最初のフレームを描くまでウィンドウを隠す）:
  <https://github.com/emilk/egui/pull/2279>

## 決定

**ルートビューポートを「不可視のホスト」にし、設定ウィンドウとログウィンドウをどちらも
子ビューポート（deferred viewport）にする。**

ホストは eframe が表示しても実害のないウィンドウとして作る。

- 1×1 ピクセル
- 装飾なし（`with_decorations(false)`）
- タスクバーに出さない（`with_taskbar(false)`）
- 画面外に配置（`with_position([-32000.0, -32000.0])`）
- `with_visible(false)`（eframe に上書きされるが、初回フレームまでの分は効く）

ホストは画面に何も描かず、毎フレーム「開いているウィンドウを宣言する」ことだけを行う。
どのウィンドウも開いていなければ再描画を要求しないので、GUI スレッドは眠る
（ADR 0035 と同じ性質。チェックリスト A-17c・B0-7）。

子ビューポートは宣言をやめれば破棄され、また宣言すれば作り直される。したがって
「閉じる＝隠す」の作り込み（`CancelClose` + `Visible(false)`）は子ウィンドウには不要になり、
`close_requested` を見てフラグを下ろすだけでよくなる。ADR 0032 が守っていた
「イベントループを終わらせない」という本質はホストが担う。

子ビューポートのコールバックは `Fn + Send + Sync` を要求されるため、ウィンドウの状態は
`Arc<Mutex<..>>` に置く。

## 理由

- **ちらつきを原理的に消せる**。対症療法では 1 フレーム分の表示が残る。eframe が必ず表示する
  ウィンドウを「表示されても見えないもの」にすれば、そもそも見えるものがない
- **無音起動（ADR 0029）を守れる**。トレイから明示的に開くまで、利用者に見えるウィンドウが
  1 つも存在しない状態を保証できる
- **構成が対称になる**。設定とログが同じ扱い（どちらも子）になり、「片方だけルートで特別扱い」
  という非対称さが消える。どちらを先に開いても挙動が同じになる
- **閉じる処理が単純になる**。子は破棄してよいので、隠す・再表示するという状態を持たずに済む

## 却下した代替案

- **次のフレームで `Visible(false)` を送り直す**（B0 で試した対症療法）: 1 フレーム分の
  表示が必ず残る。実機で「一瞬開く」と確認された
- **ルートを画面外に置いたまま設定ウィンドウとして使う**: 開くときに画面内へ移動させればよいが、
  位置とサイズを自前で計算する必要があり、マルチモニターや DPI 変更を考えると面倒が増える。
  「ウィンドウサイズと位置は記憶しない」（オーナー決定 2026-07-21）方針とも噛み合わない
- **GUI スレッドをアプリ起動時から動かし、初回フレームを起動時に済ませる**: ちらつきが
  起動時に移るだけで、無音起動を壊す

## 残るリスク

1×1・画面外・タスクバーなしのホストウィンドウが、環境によっては一瞬でも見える可能性は
ゼロではない。受け入れチェックリスト B0-11・B0-12 で、起動直後とログ単独表示時に
ウィンドウが現れないことを確認する。
