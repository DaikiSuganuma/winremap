# ADR 0031: `src/notify.rs` を unsafe 隔離リストに追加する

- ステータス: 承認（オーナー決定 2026-07-20）
- 日付: 2026-07-20
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

ADR 0029 で、起動時のコンソールウィンドウを廃止し、(1) ターミナルから起動されたときだけ親コンソールへ出力する、(2) コンソールが無い場合の重要な通知はダイアログで出す、と決めた。

これらに使う `AttachConsole` / `GetStdHandle` / `SetStdHandle` / `CreateFileW` / `MessageBoxW` はすべて Windows API であり、windows-rs では `unsafe fn` のため呼び出しに `unsafe` ブロックが必要になる。しかし不変条件 3（unsafe の隔離）が許可するのは `hook.rs` / `sender.rs` / `window.rs` / `src/ime_indicator/` / `examples/ime_probe.rs` だけで、「起動時のコンソール接続」「ユーザーへの通知ダイアログ」はどの役割にも当てはまらない。

ADR 0009 は同種の衝突（メッセージループ・named mutex を main.rs に置けない問題）を「役割として自然に説明できる既存ファイルへ寄せる」ことで解決したが、今回は寄せ先が無い。コンソール接続はフック機構とも送出とも前面ウィンドウ監視とも無関係であり、`hook.rs` に置くのは ADR 0009 の理屈（フックの完全性を守る機構だからフック側に置く）を捻じ曲げることになる。

ADR 0029 でオーナーは「ターミナル出力はデバッグ時に有用なので機能を維持し、専用ファイルを unsafe 許可リストに追加する」方針を承認済み。本 ADR はその具体化（ファイル名の確定と規約文書の改訂）を行う。

## 決定

**`src/notify.rs` を新設し、不変条件 3 の unsafe 隔離リストに追加する。**

`notify.rs` の責務は「タスクトレイ以外の手段でユーザーに伝えること」に限定する。

| 機能 | 用途 |
|---|---|
| `attach_parent_console()` | 起動元プロセスのコンソールに接続する（`AttachConsole(ATTACH_PARENT_PROCESS)`）。標準ハンドルが未設定の場合のみ `CONOUT$` を開いて設定する |
| `has_console()` / `console_line()` | 出力先が存在するときだけ印字する |
| `error()` / `info()` | コンソールがあれば印字、無ければ `MessageBoxW` でダイアログ表示 |

キー入力の処理経路（フック・送出・前面監視）には一切関与しない。

これに伴い次を改訂する。

- `AGENTS.md` 不変条件 3
- `docs/01_project-brief.md` §5-3 およびモジュール表

## 理由

- **監査面積が増えない**: 追加される unsafe は起動時に 1 回動く数行と、ダイアログ表示 1 か所のみ。いずれもキー処理経路の外にあり、フックの安定性（優先順位の第 1 位）に影響しない
- **責務が明確**: 「ユーザーへの通知手段」という 1 つの軸で説明でき、将来ここに何を足してよいかの判断がぶれない
- **既存ファイルの意味を壊さない**: ADR 0009 で `hook.rs` に寄せたものは「フックの完全性を守る機構」という一貫した説明があった。コンソール接続にその説明は成立しないため、無理に寄せるほうが規約としては劣化する
- **リダイレクトの尊重が 1 か所に閉じる**: 「標準ハンドルが既に設定済みなら上書きしない（`winremap --help > out.txt` を壊さない）」という細かい判断を 1 ファイルに閉じ込められる

## 却下した代替案

- **`AttachConsole` をやめてターミナル出力を捨てる**: 例外は不要になるが、`--debug` / `--help` をターミナルで確認できなくなる。デバッグ時の有用性が失われる点をオーナーが不可とした（ADR 0029）
- **安全ラッパーを提供する既存クレートを使う**: 依存を増やす判断が別途必要になるうえ、`MessageBoxW` 側は結局どこかに置く必要がある。数行の Win32 呼び出しのために依存を足す価値がない（ADR 0009 で single-instance クレートを却下したのと同じ理由）
- **`hook.rs` に寄せる**: 上記のとおり ADR 0009 の理屈が成立せず、`hook.rs` の責務が曖昧になる
- **main.rs にも unsafe を許可する**: 不変条件 3 の意図（レビュー面積の最小化）を最も損なう。ADR 0009 でも却下済み

## 影響・補足

- unsafe 許可ファイルは 4 → 5 になる（`hook.rs` / `sender.rs` / `window.rs` / `src/ime_indicator/` / `notify.rs`。加えて検証用 `examples/ime_probe.rs`）
- 各 `unsafe` には従来どおり `// SAFETY:` コメントを付す
- ADR 0029 のログ表示ウィンドウを egui（ADR 0030）で実装できれば、そちらで新たな unsafe は増えない見込み
