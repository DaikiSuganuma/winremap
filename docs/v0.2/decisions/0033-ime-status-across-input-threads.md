# ADR 0033: IME 状態は前面アプリの入力スレッドすべてに問い合わせる

- ステータス: accepted
- 日付: 2026-07-21
- 作成: Claude Code（AI モデル: claude-opus-4-8）／レビュー・承認: オーナー

## 文脈

オーナーの実機確認で「Windows 11 のメモ帳では IME インジケーターが表示されない。
他のアプリでは表示された」という報告があった。

実機（Windows 11 Pro 26200、メモ帳 11.2605.34.0）でメモ帳のウィンドウツリーを
ダンプしたところ、原因が特定できた。IME がオンの状態で次の結果になる。

```
hwnd=310a8  tid=27004  cls=Notepad                    ime=5109c   open=0
  hwnd=321052 tid=6052  cls=NotepadTextBox             ime=770c8a  open=1
  hwnd=2e0b74 tid=6052  cls=RichEditD2DPT              ime=770c8a  open=1
  ...（他の子ウィンドウはすべて tid=27004、ime=5109c、open=0）
```

Windows 11 のメモ帳は WinUI 3 アプリで、**フレームウィンドウ（スレッド 27004）と
テキスト編集部（スレッド 6052）が別の UI スレッド**にある。`ImmGetDefaultIMEWnd` は
**ウィンドウが属するスレッド**の既定 IME ウィンドウを返すため、前面ウィンドウ
（フレーム）を起点にすると常に編集部とは別のスレッドを見ることになり、
`IMC_GETOPENSTATUS` は必ず 0（オフ）を返していた。

ADR 0023 で対処した UWP の CoreWindow 問題も本質は同じ（入力の実体が前面ウィンドウ
とは別のウィンドウにある）だが、あちらはクラス名決め打ちの特別扱いだったため
WinUI 3 には効かなかった。

参照:

- `ImmGetDefaultIMEWnd`（スレッド単位の既定 IME ウィンドウを返す）:
  <https://learn.microsoft.com/en-us/windows/win32/api/imm/nf-imm-immgetdefaultimewnd>
- `WM_IME_CONTROL` / `IMC_GETOPENSTATUS`:
  <https://learn.microsoft.com/en-us/windows/win32/intl/wm-ime-control>
- `EnumChildWindows`:
  <https://learn.microsoft.com/en-us/windows/win32/api/winuser/nf-winuser-enumchildwindows>

## 決定

**前面ウィンドウとその子孫ウィンドウを走査し、重複を除いた既定 IME ウィンドウすべてに
問い合わせる。1 つでもオンを返したらオンとする。**

- 前面ウィンドウ自身を最初に問い合わせる。単一スレッドの通常アプリは既定 IME ウィンドウが
  1 つしかないため、v0.1 とまったく同じ 1 回の問い合わせで完結する
- 子孫の走査は `EnumChildWindows`。既定 IME ウィンドウは GUI スレッドごとに 1 つなので、
  重複除去すると問い合わせ回数は「前面アプリの GUI スレッド数」になる（メモ帳で 2）
- 上限は子ウィンドウ 128 個・IME ウィンドウ 4 個。最悪でも
  4 × `QUERY_TIMEOUT_MS`（100ms）= 400ms で必ず返る
- 収集は固定長配列で行い、ヒープ確保をしない
- ADR 0023 の CoreWindow 特別扱いは削除した。子孫走査が CoreWindow も拾うため、
  一般解に吸収される
- `--debug` の表示は「CoreWindow 経由」から「子ウィンドウ経由」に変更した

この処理はインジケータースレッドで動き、フックコールバックには一切関係しない
（不変条件 2）。問い合わせはトグルキー・前面変更をきっかけとしたイベント駆動であり、
定期ポーリングではないため、回数増加のコストは実質無視できる。

## 理由

- **クラス名を増やし続ける保守を避ける**。`Windows.UI.Core.CoreWindow` に
  `Microsoft.UI.Content.DesktopChildSiteBridge` や `RichEditD2DPT` を足していく方式は、
  新しい UI フレームワークが出るたびに破綻する。「アプリのどの入力スレッドがオンか」を
  直接見るほうが原理に近い
- **通常アプリの挙動が変わらない**。既定 IME ウィンドウが 1 つなら問い合わせも 1 回で、
  v0.1 と同一である
- **失敗しても表示しないだけ**（設計書 §3.2）。不変条件 1 への影響はない

## 却下した代替案

- **`GetGUIThreadInfo(0).hwndFocus` を使う**: 原理的には最も正確だが、返るのは
  「前面スレッドのフォーカスウィンドウ」であり、メモ帳のように前面ウィンドウと編集部が
  別スレッドの場合に編集部を指す保証がない。実機で確認しようとしたが、メモ帳を前面に
  していない状態では `hwndFocus` が 0 になるため検証できず、確証のない方式は採らなかった
- **`RichEditD2DPT` / `DesktopChildSiteBridge` をクラス名で特別扱いする**: 対症療法であり、
  上記のとおり保守が続かない
- **可視の子ウィンドウだけを走査する**: 誤検知（フォーカスのないスレッドの古いオン状態を
  拾う）を減らせる可能性があるが、UWP の CoreWindow が可視かどうかを実機で確認できず、
  ADR 0023 の回帰を招きかねないため見送った

## 残るリスク

「アプリ内に IME 入力を持つスレッドが複数あり、片方がオン・片方がオフ」の場合、オンが
勝つため実際にフォーカスしている側がオフでもパネルが出る可能性がある。インジケーターは
一時表示のヒントであり、表示過多より無表示のほうが問題（報告された不具合そのもの）と
判断して許容する。受け入れチェックリストでメモ帳の**オフ状態**も確認する。
