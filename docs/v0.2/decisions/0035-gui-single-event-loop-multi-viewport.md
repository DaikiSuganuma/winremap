# ADR 0035: 設定 GUI とログウィンドウを 1 つの eframe アプリの複数ビューポートで動かす

- ステータス: accepted
- 日付: 2026-07-21
- 作成: Claude Code（AI モデル: claude-opus-4-8）／レビュー・承認: オーナー（2026-07-21）

## 文脈

Phase A でトレイの「ログを表示」を eframe で実装した結果、**winit のイベントループは
1 プロセスに 1 つしか作れない**という制約が判明した（[ADR 0032](0032-log-window-hide-instead-of-close.md)）。
ログウィンドウは既にそのイベントループを専用スレッドで持っている。

設定 GUI（v0.2 の目玉機能）はもう 1 つのウィンドウを必要とするため、
「2 つ目の `eframe::run_native` を呼ぶ」ことはできない。同居のさせ方を決める必要がある。

## 決定

**1 つの eframe アプリが 2 つのビューポート（ウィンドウ）を持つ構成にする。**

- **ルートビューポート = 設定 GUI ウィンドウ**。`eframe::run_native` が作る唯一のウィンドウであり、
  閉じることはできない（ADR 0032 と同じく、閉じる操作は非表示化）
- **子ビューポート = ログウィンドウ**。`Context::show_viewport_deferred` で作る
  [deferred viewport](https://docs.rs/egui/0.35.0/egui/struct.Context.html#method.show_viewport_deferred)。
  親と別に再描画されるため、ログの更新が設定画面の描画を巻き込まない
- 起動直後は**両方とも非表示**。トレイの「設定」「ログを表示」でそれぞれ表示する
- GUI スレッドは 1 本のまま。フックのメッセージループ（メインスレッド）には触れない
- eframe / winit のバージョンは引き続き `Cargo.toml` で固定する。`winit` を直接依存に持つのは
  `with_any_thread(true)` のためだけであり、eframe が使う版と一致させる必要がある

`src/log_window.rs` は「ログウィンドウ 1 枚を動かすモジュール」から
「GUI スレッドを持ち、その上で複数のウィンドウを動かすモジュール」へ一般化する。
モジュール分割は設計書（[04_config-gui-design.md](../04_config-gui-design.md)）で確定する。

### 実装上の注意（先に潰すべき点）

deferred viewport は**親が毎フレーム `show_viewport_deferred` を呼び続ける限り生存する**。
つまりログウィンドウを開いている間は、非表示のルート（設定 GUI）もフレームを回す必要がある。
eframe 0.35 は不可視ウィンドウを直接描画する経路を持ち（ADR 0032 で確認済み、egui Issue #5229）、
その再描画は 100ms にスロットルされるため、コストは小さい。ルートの `ui` は
「どちらかのウィンドウが表示中なら `request_repaint_after` を呼ぶ」ことにする。

両方とも非表示のときは何も要求しない。GUI スレッドは眠り、常駐コストはスレッド 1 本のみになる
（ADR 0032 と同じ性質。チェックリスト A-17c で確認済みの挙動を維持する）。

**この構成はまだ実機で動かしていない。** 設定 GUI の中身を作る前に、
「ルート＋子ビューポートの開閉」だけを実装して Phase A のログウィンドウ項目
（A-10〜A-20）を再確認する（設計書のマイルストーン B0）。

## 理由

- **プロセスが 1 つで済む**。多重起動防止（名前付き mutex）、設定のリロード連携、終了処理が
  すべて既存のしくみのまま使える。プロセス間通信を自作しないで済むのは「単純さ」の優先順位に合う
- **egui の標準機能に乗る**。deferred viewport は egui/eframe が公式に持つマルチウィンドウ機構であり、
  自前でイベントループを分岐させるより壊れにくい
- **フックへの影響が変わらない**。GUI スレッドが 1 本という点は Phase A と同じで、
  A-12（ログウィンドウを開いたままリマップが遅延しない）の確認結果がそのまま効く

## 却下した代替案

- **設定 GUI を別プロセスにする**（`winremap.exe --config-gui`）: 本体の安定性は最も守られ、
  イベントループ制約も回避できる。しかし多重起動防止・保存後のリロード通知・GUI 側での設定パス
  解決をプロセス間で作り込む必要があり、v0.2 の規模に対して機構が重い。将来 GUI 側の不安定さが
  問題になったら、この案へ移行する余地は残る（本 ADR を新 ADR で覆す）
- **ウィンドウを 1 枚にし、ログを設定 GUI 内のタブにする**: 最も単純だが、「ログだけ見たい」ときに
  設定画面が開くことになり、Phase A で作ったログウィンドウの独立性（トラブル時に軽く開ける）が
  失われる。オーナー判断で却下
- **`show_viewport_immediate` を使う**: 実装は簡単だが、親と子が互いの再描画を強制し合う仕様であり、
  ログが流れるたびに設定画面まで再描画されてしまう
