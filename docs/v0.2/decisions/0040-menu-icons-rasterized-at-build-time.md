# ADR 0040: トレイメニューのアイコンは Bootstrap Icons をビルド時にラスタライズする

- ステータス: accepted
- 日付: 2026-07-21
- 作成: Claude Code（AI モデル: claude-opus-4-8）／レビュー・承認: オーナー（2026-07-21）

## 文脈

オーナーの要望: トレイメニューの各項目の左にアイコンを表示して分かりやすくしたい。
アイコンは [Bootstrap Icons](https://icons.getbootstrap.com/) を使いたい。

技術的な制約が 2 つある。

1. **Win32 のメニューは生のピクセル（RGBA ビットマップ）しか受け付けない。** `muda` の
   `IconMenuItem` が取るのも `Icon::from_rgba` である。SVG を直接渡す経路は無い
2. **Bootstrap Icons は SVG と WOFF/WOFF2 でしか配布されていない。** TTF/OTF が無いため、
   フォントとして読み込む手も使えない

つまり、どこかで SVG → ピクセルの変換が必要になる。

## 決定

**`build.rs` でビルド時に SVG をラスタライズし、生成した RGBA だけを実行ファイルに埋め込む。**

- `resvg` を **build-dependencies** に入れる。実行時の依存にはならず、**実行ファイルには
  resvg のコードは 1 バイトも入らない**。入るのは 16×16×4 = 1KB × 4 個のピクセルのみ
- SVG は `assets/icons/` にベンダリングする（`gear` / `arrow-clockwise` / `card-list` /
  `box-arrow-right`）。ネットワークからの取得はビルド時にも行わない
- `fill="currentColor"` はビルド時に固定色へ置換する。Win32 のポップアップメニューは、
  アプリがダークモードに明示的に対応しない限り明るい背景で描かれるため、暗いグレー
  （`#343a40`）とする
- ラスタライズに失敗したらビルドを失敗させる。アイコンが静かに消えるほうが厄介である
- tiny-skia は乗算済みアルファで保持するため、RGBA へ戻す際に **demultiply する**
  （しないとアンチエイリアスの縁が濁る）
- アプリ名の行だけは Bootstrap Icons ではなく **WinRemap 自身のアイコン**を使う
- 「有効」（チェック項目）にはアイコンを付けない（オーナー決定）。チェックマークが
  その項目の印であり、隣に別の記号が並ぶと競合する

ライセンス表記は README の謝辞に追加する（Bootstrap Icons、MIT）。

## 理由

- **要望どおり Bootstrap Icons が使える**。素材は SVG のままリポジトリに残るので、後から
  サイズを変えたり差し替えたりできる
- **配布物への影響がほぼゼロ**。ビルド依存はランタイム依存とは別枠であり、
  「単一バイナリ・依存なし」という v0.1 からの性質を壊さない
- **egui 側にも同じ手が使える**。将来ウィンドウ内でアイコンが要るとき、`egui_extras` の
  SVG 機能（resvg をランタイムに引き込む）を入れずに済む
- 実測: リリース exe は 8.20MB → 8.40MB。増分の大半は eframe のリビルド差分で、
  アイコン自体は 4KB である

## 却下した代替案

- **`egui_extras` の svg 機能を使う**: ランタイムに resvg + usvg + tiny-skia が入り、
  依存も実行ファイルも増える。しかもトレイメニューは egui ではないので、これでは解決しない
- **Bootstrap Icons のフォントを埋め込む**: TTF/OTF 配布が無く、egui は WOFF2 を読めない
- **PNG を手で書き出してアセットに置く**: 変換工程は不要になるが、素材の生成が手作業になり、
  サイズ変更のたびに書き出し直す必要がある
- **Unicode の記号で代用する**（`⚙` `↻` など）: 依存ゼロで最も軽いが、字形がフォント任せで
  揃わず、要望された見た目にならない。ウィンドウ内の注記（`※` / `ℹ`）はこの方式のままとする

## 残るリスク

Windows のダークモードでメニューが暗い背景で描かれる環境では、暗いグレーのアイコンが
見えにくくなる可能性がある。WinRemap はダークモードに明示的に対応していないため、
現状は明るい背景で描かれるはずだが、受け入れチェックリストでダークモードでの見え方を確認する。
問題があれば、明暗 2 種を焼き込んでシステムテーマで切り替える（テーマ判定の Win32 呼び出しが
増えるため、必要になった時点で別 ADR とする）。
