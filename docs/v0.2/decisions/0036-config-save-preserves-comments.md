# ADR 0036: 設定の保存は toml_edit でコメント・書式を保持した部分更新にする

- ステータス: accepted
- 日付: 2026-07-21
- 作成: Claude Code（AI モデル: claude-opus-4-8）／レビュー・承認: オーナー（2026-07-21）

## 文脈

設定 GUI（Phase B）は `%APPDATA%\winremap\config.toml` を書き換える。v0.1 の読み込みは
`toml` クレートで構造体へデシリアライズする一方向であり、書き戻す経路は存在しない。

WinRemap の設定は手書き前提で育ってきた。`examples/suganuma.toml` のような実運用の設定は
コメントで意図を残し、キーマップを意味のある順に並べている。GUI で 1 か所いじっただけで
それが消えるのは、既存利用者にとって受け入れがたい。

## 決定

**保存は [`toml_edit`](https://docs.rs/toml_edit/) による部分更新とし、コメント・並び順・空行・
インデントを保持する。**

- 読み込み（バリデーション・キーマップ構築）は従来どおり `toml` クレートと `src/config/` の
  ロジックを使う。GUI のためにこの経路を変えない
- GUI は保存時に、元ファイルを `toml_edit::DocumentMut` として読み直し、**変更された項目だけ**を
  書き換えて保存する。触っていない箇所は 1 バイトも動かさない
- 保存前に、書き上がった TOML を既存の `config::load` 相当で**必ず検証する**。検証に失敗する
  内容は保存しない（壊れた設定を書き出さない）
- 保存は**原子的に行う**。同ディレクトリの一時ファイルへ書いてから置き換える。書き込み途中の
  クラッシュで設定ファイルを失わないため
- `toml_edit` のバージョンは実装時点の最新（0.25.13、ライセンス MIT OR Apache-2.0）を
  `Cargo.toml` で固定する

## 理由

- **既存利用者の設定を壊さない**。これが最大の理由である。GUI は「設定ファイルを置き換えるもの」
  ではなく「設定ファイルを編集する道具」であり、手書きとの併用が前提になる
- **GUI で扱わない項目を失わない**。将来 TOML に項目が増え、GUI がまだ対応していない期間があっても、
  部分更新なら未知のキーはそのまま残る。丸ごと書き直す方式では、GUI が知らない設定は保存のたびに
  消える
- **検証を通してから書くので、GUI 経由で壊れた設定が生まれない**。v0.1 の「壊れた設定をリロードしても
  直前の設定を維持する」方針（設定仕様 §4）と一貫する
- 依存が 1 つ増えるが、`toml_edit` は `toml` クレート自身が土台にしている定番であり、
  ライセンスも MIT OR Apache-2.0 で方針に合う

## 却下した代替案

- **構造体から TOML を丸ごと生成し直す**: 実装は最も単純だが、保存のたびにコメントと並び順が消える。
  オーナーの設定運用と相容れない
- **GUI からは保存しない（閲覧と検証のみ）**: 壊すリスクはゼロだが、「設定 GUI を v0.2 の目玉に
  する」という位置づけを満たさない
- **設定ファイルを GUI 専用の形式（JSON 等）に分離する**: TOML を正とする現行仕様（設定仕様 §1）を
  壊し、手書き利用者を切り捨てることになる

## 残るリスク

`toml_edit` での部分更新は「どこを触ったか」を GUI 側が正確に追える必要がある。キーマップの
並べ替えや削除では、周辺のコメントがどちらの項目に属するかが曖昧になりうる。設計書
（[04_config-gui-design.md](../04_config-gui-design.md)）で、削除・並べ替え時のコメントの扱いを
明記し、受け入れチェックリストでコメント付き設定の往復を確認する。
