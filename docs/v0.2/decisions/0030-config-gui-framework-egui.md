# ADR 0030: 設定 GUI のフレームワークに egui を採用する

- ステータス: 承認（オーナー決定 2026-07-20）
- 日付: 2026-07-20
- 作成: Claude Code（AI モデル: claude-fable-5）

## 文脈

v0.2 の目玉機能である設定 GUI（[01_development-plan.md](../01_development-plan.md) Phase B）と、ADR 0029 で決めたトレイの「ログ表示」ウィンドウを、どの GUI フレームワークで実装するかを決める必要がある。

オーナーが **gpui-component**（Zed の gpui 上のコンポーネントライブラリ）を検討していたため、egui・Slint と比較調査を行った。調査結果は [02_gui-framework-study.md](../02_gui-framework-study.md)（全項目に出典 URL 付き）にまとめてある。

WinRemap 側の制約は次のとおり。

- ライセンスは MIT / Apache-2.0 系が望ましい（本体は MIT）
- 日本語 UI 必須（ADR 0014）
- 単一 exe 配布を維持し、外部ランタイム（WebView2 等）に依存しない
- 既存のタスクトレイ（`tray-icon` クレート、ADR 0007）およびキーフックのメッセージループと共存すること
- 優先順位は **安定性（フックを止めない）> 単純さ > 機能の豊富さ**

## 決定

**egui（eframe または egui-winit）を採用する**。まず egui で進め、設定 GUI とログ表示ウィンドウの両方をこれで実装する。

日本語表示のために、フォントの読み込み方式（システムフォント読込 or 埋め込み）を実装時に決める。

## 理由

- **ライセンス**: `MIT OR Apache-2.0` で本体の MIT と整合し、帰属表示義務もない。3 候補で唯一、要件に完全合致する
- **既存資産との整合**: `tray-icon` クレートに公式の egui サンプル（`examples/egui.rs`）があり、ADR 0007 で採用済みのトレイ実装をそのまま活かせる。トレイ常駐との組み合わせが実証されているのは 3 候補中 egui だけだった
- **配布**: crates.io で完結するためバージョン固定と再現ビルドができる。バイナリ増分も約 5MB と最小クラスで、外部ランタイム依存もなく単一 exe 配布を維持できる
- **用途との相性**: 設定 GUI は「たまに開く単純なフォーム」であり、即時モードなら UI と設定データの状態同期コードが不要になる。GUI 層を薄く保てる点が「単純さ」の優先順位に合う

## 却下した代替案

- **gpui-component**: ライセンス（Apache-2.0）と見た目の完成度は良いが、(1) 公式のインストール手順が git 依存を指示しており crates.io でのバージョン固定ができない、(2) `Application::run()` がメインスレッドを取る設計で、**タスクトレイおよびキーフックのメッセージループと共存できるかの実績が確認できない**（不変条件 1・2 に直結するリスク）、(3) バイナリ約 12MB と最大、(4) Zed の Windows 対応は 2025 年 10 月に stable 化したばかりで日本語 IME の個別不具合報告がある。設定ダイアログ程度の UI に対して依存の重さとリスクが釣り合わない
- **Slint**: 技術的完成度は最も高く（1.x 安定版、日本語フォント自動選択、`run_event_loop_until_quit()` がトレイ常駐を公式に想定）、この用途には最も素直だった。しかし無償利用は GPL-3.0-only（本体 MIT からの実質的なライセンス変更）か Royalty-free 2.0（About 画面等での帰属表示義務）のいずれかで、MIT/Apache 系という要件から外れるため却下
- **iced**: Elm 風の保持モード。egui に対する決定的な利点がなく、バイナリも約 11MB と大きい
- **Tauri（WebView ベース）**: WebView2 ランタイム依存が単一 exe 配布の方針に反する
- **ネイティブ Win32**: 依存ゼロだが開発コストが大きく、何より `unsafe` が大量に増えて不変条件 3（unsafe の隔離）と衝突する

## 影響・補足

- egui は 1.0 未到達で、リリースごとに破壊的変更がある。`Cargo.toml` でバージョンを固定し、更新時は CHANGELOG を確認する
- **実装前に潰すべきリスク（プロトタイプで検証する）**:
  1. **日本語フォント**: 既定フォントに CJK グリフが同梱されていないため、`FontDefinitions` への登録が必須。システムフォント（`C:\Windows\Fonts\meiryo.ttc` 等）の読込か、フォント埋め込み（バイナリが数 MB 増える）かを選ぶ
  2. **キーフック・トレイとの共存**: `eframe::run_native` は既存のイベントループ上では動かせない（egui Issue #2875）。ウィンドウを閉じてトレイに戻し再度開く挙動を実現するには `egui-winit` + `egui-wgpu` で自前の winit ループを持つ構成が必要になる可能性がある。**キーフックを止めないこと（不変条件 1・2）が最優先**であり、GUI を別スレッド・別プロセスのいずれで動かすかも含めて Phase B の設計書（`03_config-gui-design.md`）で確定させる
  3. **日本語 IME**: Windows での日本語入力の実績が確認できていない。設定名などのテキスト入力欄を設ける場合は早期に実機検証する
- ADR 0029 のログ表示ウィンドウも egui で実装できれば、自前の Win32 呼び出し（= 新たな `unsafe`）を増やさずに済む
