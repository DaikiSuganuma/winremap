# ADR 0039: マクロ設定を `[macro] delay_ms` に移す（旧 `macro_delay_ms` も受け付ける）

- ステータス: accepted（[ADR 0019](../../v0.1/decisions/0019-macro-delay-in-config.md) のキー名を差し替える）
- 日付: 2026-07-21
- 作成: Claude Code（AI モデル: claude-opus-4-8）／レビュー・承認: オーナー（2026-07-21）

## 文脈

v0.1 では、マクロの打鍵間隔をトップレベルの `macro_delay_ms` で指定していた（ADR 0019）。
一方 IME インジケーターは `[ime_indicator]` というセクションを持つ。設定 GUI の「全体設定」で
両者を並べたところ、片方だけがセクションを持たず、表示の形が揃わないことが目立った。

オーナー判断（2026-07-21）: **マクロ設定も `[macro]` セクションに入れ、`delay_ms` にする。**

## 決定

```toml
[macro]
delay_ms = 8  # マクロの各打鍵の間隔（ms、0-15）
```

- **旧形式 `macro_delay_ms` も引き続き受け付ける。** v0.1 で書かれた設定ファイルが、
  更新しただけで壊れることがあってはならない
- 両方が書かれていた場合は**検証エラー**とし、どちらが効いているか分からない状態を作らない。
  行番号付きで「`[macro] delay_ms` だけを残してください」と報告する
- 範囲（0-15）とその意味は ADR 0018／0019 のまま変わらない。`--macro-delay` の CLI 上書きも同じ
- 設定 GUI は `[macro]` セクションとして表示し、`[ime_indicator]` と同じ 4 列の表
  （設定・キー・値・コメント）で並べる

`examples/suganuma.toml` は新形式へ更新した。他の例（`minimal.toml`、`emacs.toml`）は
この設定を使っていない。

## 理由

- **設定ファイルの見た目が揃う**。トップレベルのばら撒きキーとセクションが混在するより、
  「機能ごとにセクション」のほうが増えたときに読みやすい。将来マクロ関連の設定
  （最大長など）が増えても収まる場所がある
- **設定 GUI の構造がそのまま設定ファイルの構造になる**。GUI はセクション単位で表を作るため、
  ファイル側にセクションがあるほうが対応が素直である
- **既存利用者を切らない**。旧キーを受け付け続けるコストは小さく、v0.1 の設定が動き続ける
  価値のほうがはるかに大きい

## 却下した代替案

- **旧形式を廃止する（破壊的変更）**: v0.1 の設定ファイルが読めなくなる。バージョンを上げた
  だけでリマップが止まるのは、常駐ツールとして最悪の体験である
- **両方書かれたら新形式を優先し、黙って続行する**: 動きはするが、利用者は自分の設定の
  どちらが効いているかを知る機会を失う。設定エラーは全件まとめて報告するという方針
  （設定仕様 §4）とも合わない
- **`macro_delay_ms` のままにして GUI 側だけ整える**: GUI に「セクションが無い設定」の
  特別扱いが残り、B2 以降の編集・保存でも同じ特別扱いが必要になる

## 移行

利用者が何もしなくても動き続ける。README・ヘルプサイト・設定仕様は新形式で記述し、
旧形式は「v0.1 の書き方も引き続き有効」と併記する。
