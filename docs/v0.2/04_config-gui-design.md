## 1. 目的

設定ファイルを手書きしなくても、**トレイから設定を開いて編集・保存し、そのまま反映できる**
ようにする。同時に、手書き派の設定を壊さないこと（ADR 0036）を絶対条件とする。

キー記法（`C-h`、`A-x h`、`["C-Right", "C-Left"]`）は Keyhac / fakeymacs 経験者には自然でも、
新規利用者には最初の壁になる。GUI の主な価値は「**何が書けるかを見せること**」であり、
`application` に何を書けばよいか、どのキー名が使えるか、どこが間違っているかを、
ドキュメントを開かずに知れる状態にする。

## 3. 画面構成

単一ウィンドウ。左に一覧、右に詳細の 2 ペイン構成とする。

```
┌─ WinRemap 設定 ──────────────────────────────────────────────┐
│ [保存] [元に戻す] [再読み込み]            設定ファイル: …\config.toml │
├──────────────────┬───────────────────────────────────────────┤
│ キーマップ          │ ■ キーマップ「emacs」                       │
│  ▸ notepad        │   名前        [emacs                  ]    │
│  ▸ emacs      ◀   │   対象アプリ   [chrome.exe] [code.exe] [+]  │
│  ▸ *（全アプリ）    │   除外        （* のときのみ）              │
│  [＋ 追加] [－ 削除] │                                           │
│                  │   リマップ規則                              │
│ 全体設定           │   ┌─────────────┬──────────────┬───┐      │
│  ▸ 基本            │   │ 入力         │ 出力          │   │      │
│  ▸ IME インジケーター│   │ C-h         │ Back         │ － │      │
│                  │   │ A-x h       │ C-z          │ － │      │
│                  │   │ C-t         │ C-Right, C-Left│ － │      │
│                  │   └─────────────┴──────────────┴───┘      │
│                  │   [＋ 規則を追加]                           │
├──────────────────┴───────────────────────────────────────────┤
│ ⚠ 12 行目: 未知のキー名 "Bak"（Back の誤り？）                    │
└──────────────────────────────────────────────────────────────┘
```

### 3.1 一覧ペイン（左）

- `[[keymap]]` を定義順に並べる。`application = ["*"]` は「\*（全アプリ）」と表示する
- 追加・削除・並べ替え（ドラッグではなく上下ボタン。実装が単純で、誤操作も少ない）
- 「全体設定」はキーマップと同じ一覧に置き、選ぶと右ペインが全体設定になる。
  画面をもう 1 つ作るより移動が少ない

### 3.2 詳細ペイン（右）

- **名前** — 単一行テキスト
- **対象アプリ** — exe 名のチップ列。`[+]` を押すと入力欄が出る。
  **「今の前面アプリから取得」ボタン**を併設する（後述 §3.4）
- **除外** — `application` が `*` のときだけ表示する
- **リマップ規則** — 入力・出力の表。行の追加・削除。
  入力欄・出力欄はテキスト入力とし、**キー記法をそのまま書く**（§3.3）

### 3.2.1 記法ペイン（右端・キーマップ選択時のみ）

`C-` / `A-` などの意味を示す凡例は、規則の表の下ではなく**右端の独立したペイン**に置く
（Windows エクスプローラーの詳細ウィンドウと同じ位置づけ。オーナー決定 2026-07-21）。
規則を上から下へ読んでいる最中に、目線を大きく動かさず参照できる。

全体設定を選んでいるときは表示しない。説明する対象が無いためである。

### 3.2.2 セクション見出し

各セクションの見出しには Bootstrap Icons を付ける（対象アプリ・除外アプリ・リマップ規則・
マクロ・IME インジケーター・キー記法）。素材は白で焼き、描画時に本文色へ着色するため、
明暗テーマの両方に追従する（[ADR 0042](decisions/0042-window-icons-white-and-tinted.md)）。

### 3.2.2a ログウィンドウの操作系

上部は表示の見え方を変えるもの（「最新に追従」）だけとし、「消去」「全体をコピー」は
フッターへ置く。ログを読むことがこの窓の目的であり、最新行の近くにボタンを置かない
（オーナー決定 2026-07-21）。

### 3.2.2b ヘッダー（設定ファイルの情報）

設定ファイルのパス・更新日時・読み込み日時は 2 列（項目／値）の表で示す。
更新日時と読み込み日時は互いに読み比べて初めて意味を持つためで、
「保存したが再読み込みしていない」状態はこの窓が他に示す手段を持たない。

外部アプリを開くリンクにはアイコンを付ける（テキストエディタで開く =
`box-arrow-up-right`、ヘルプページ = `link-45deg`）。WinRemap の外へ出ることを
押す前に分かるようにするためである。

「テキストエディタで開く」と「設定を再読み込み」は**注記の下**に置く。
「この窓は見るだけ → エディタで変える → ボタンで反映する」という順に読めるためである。

### 3.2.3 リマップ規則の表

列は「WinRemap 入力」「アプリ向け出力」「コメント」。単に「入力／出力」としないのは、
どちら側が自分の打鍵でどちら側がアプリに届く結果なのかを取り違えやすいためである。

**同じ入力を他のキーマップも束縛している場合は「他のキーマップ」列を追加する。**
実際に効くのは 1 つだけ（アプリ指定 > `*`、同種なら先勝ち。ADR 0004）だが、
設定ファイルを読んでいるだけでは重複に気づけない。列は該当が 1 件も無ければ出さない。

### 3.2.4 表の体裁

対象アプリ・除外アプリ・リマップ規則・全体設定はいずれも同じ体裁の表とする
（`config_window.rs` の `table()` に集約）。

- 見出し行は**地と文字の色を反転**する。egui の `Grid` に見出しの概念が無いため、
  縞模様と同じ `with_row_color` で 0 行目だけ別色にしている（セル単位で塗ると
  帯にならない）
- 表全体を細い罫線（`Frame` の `stroke`）で囲む
- セルの余白は 4px。`Grid` の `spacing` は隣接セルの間隔なので、片側 4px になるよう与える
- 表の罫線とセルの間は 8px。文字が罫線に接すると読みにくい。左右で同じ値にするのは、
  見出しの帯が枠の中央に収まるようにするため
- 注記と表の間は 8px 空ける。詰まっていると表の一部に見える
- セクションの区切り線は上下に 20px 空け、線自体は既定の `ui.separator()` より薄くする。
  ウィジェットの枠と同じ濃さでは、文章の区切りとしては強すぎる
- 表の横幅はペインに追従させ、最終列（コメント）は折り返す。`Grid` はセル幅を
  制限しないため、長いコメントは指定しない限り折り返さずに表を画面外へ押し出す
- ボタン内の余白は上下 8px・左右 16px（egui の既定 6x2 では文字とほぼ同じ高さになる。
  左右を厚くするのは、横に詰まって見えるほうが先に目につくため）。
  `Style::spacing::button_padding` でウィンドウ全体に効かせる
- ボタンとリンクの文字の前にはアイコンを置く。チェックボックスには置かない
  （チェックマークがその項目の印であり、隣に別の記号が並ぶと競合する。
  トレイの「有効」と同じ判断。ADR 0040）
- 設定ウィンドウの初期サイズは 960x860。ウィンドウサイズは記憶しない方針
  （オーナー決定 2026-07-21）なので、毎回この大きさで開く

### 3.3 キー記法の入力

キー記法はテキストで入力し、**入力中に随時検証して欄の直下に結果を出す**。

- 妥当なら、解釈結果を人間語で表示する（例: `C-h` → 「Ctrl + h」、`A-x h` → 「Alt+x に続けて h（2 ストローク）」）
- 妥当でなければ、理由と候補を出す（例: `Bak` → 「未知のキー名。`Back` のことですか？」）
- 使えるキー名の一覧は、欄の横の `?` から一覧表示する
- 出力欄はカンマ区切りでマクロを表す（`C-Right, C-Left` → `["C-Right", "C-Left"]`）

**キーを実際に押して入力する「キーキャプチャ」は v0.2 では作らない。** WinRemap 自身がフックで
キーを掴んでいるため、キャプチャ中だけリマップを止めるなどの調停が必要になり、不変条件 1
（フックを止めない）に対するリスクが機能の価値に見合わない。将来の課題とする。

### 3.4 「今の前面アプリから取得」

`application` に何を書けばよいか分からない問題（README に「ログを表示して確認」と書いている点）を
GUI で直接解く。ボタンを押すと **3 秒のカウントダウン後**に前面アプリの exe 名を取得して欄に入れる。
その間に対象アプリへ切り替えてもらう。前面アプリの取得は `src/window.rs` の既存経路を使う。

### 3.5 全体設定

- `macro_delay_ms`（0-15）— スライダー。「マクロが取りこぼされるアプリ向け」の説明を添える
- `[ime_indicator]` — `enabled` チェックボックス、`duration_ms` / `size` / `opacity` のスライダー、
  `show_app_name` チェックボックス、`trigger_keys` のキー記法入力

## 4. 編集モデル

### 4.1 データの流れ

```
config.toml ──(toml クレート)──▶ ConfigDraft ──(GUI 編集)──▶ ConfigDraft'
     │                                                          │
     └──(toml_edit: DocumentMut)────────────────────────────────┤
                                                                ▼
                                      差分だけ DocumentMut へ適用 → 検証 → 原子的に保存
```

- **`ConfigDraft`** — GUI が編集する中間表現。`src/config` の検証済み型ではなく、
  **未検証の文字列を保持できる**（`"Bak"` のような入力途中の値を持てないと編集できない）
- 保存時のみ `DocumentMut` を触る。GUI の編集中はファイルを読み書きしない
- 保存の直前に、生成した TOML を `config::load` 相当で検証する。失敗したら保存しない

### 4.2 コメントと並び順（ADR 0036 の残るリスクへの回答）

- **値の変更**: 該当する値のみ差し替える。行のコメント（`key = "value"  # メモ`）は残る
- **項目の追加**: 対応するテーブル／配列の末尾に追加する。既存項目の間には割り込まない
- **項目の削除**: 項目とその**直前に密着したコメント行**（空行を挟まないコメント）を一緒に消す。
  空行で離れたコメントは、直前の項目に属さないものとみなして残す
- **並べ替え**: `[[keymap]]` テーブルごと、密着コメントを含めて移動する
- 上記のいずれも、保存後の差分が利用者にとって説明可能であることを基準とする。
  受け入れチェックリストでコメント付き設定の往復を確認する（§8）

### 4.3 外部変更との衝突

GUI を開いている間にテキストエディタで `config.toml` が書き換えられる場合がある。

- 保存時に、開いたときのファイルの**更新時刻とサイズ**を比較する
- 変わっていたら「ファイルが外部で変更されています。上書きしますか / 読み直しますか」と尋ねる。
  黙って上書きしない
- ファイル監視による自動リロードは v0.2 のスコープ外（開発計画 §1 で見送り済み）

### 4.4 保存とリロードの連携

保存が成功したら、**トレイの「設定を再読み込み」と同じ経路**でリロードする。
GUI 専用のリロード経路は作らない（ADR 0003 の原子的スワップをそのまま使う）。
結果はログウィンドウにも 1 行残る（Phase A で整えた経路）。

## 5. エラー表示

- **入力中**: 該当欄の直下に理由を出す。保存ボタンは押せるが、押すと保存前検証で止まる
- **保存前検証の失敗**: 画面下部のバーに件数と先頭の内容を出し、クリックで該当箇所へ移動する。
  v0.1 の「行番号付きで全件まとめて報告」を GUI でも守る（設定仕様 §4）
- **保存の失敗**（書き込み権限など）: ダイアログではなく画面下部のバーに出す。
  GUI が前面にあるのだから、モーダルにする理由がない

## 6. スレッド構成と不変条件

- GUI は Phase A で作った GUI スレッドの上で動く（ADR 0035）。**フックコールバックには触れない**
- GUI から設定を差し替える経路は既存の `hook::REMAP_TABLE` の原子的スワップのみ（ADR 0003）
- 前面アプリ名の取得は既存の `src/window.rs` 経由。GUI から Win32 を直接呼ばない
  （不変条件 3 の unsafe 隔離を維持し、GUI モジュールに unsafe を持ち込まない）
- GUI が panic しても常駐は生き続ける。GUI スレッドは `catch_unwind` で包む
  （IME インジケーターと同じ失敗隔離）
- ディスクに書くのは `config.toml` だけ。**ログは一切書かない**（不変条件 6）

## 8. テスト方針

- **編集モデルは純粋ロジックとして単体テスト**する。GUI 描画層は手動確認
  （フック層のテスト免除ルールと同様。AGENTS.md ワークフロー）
  - `ConfigDraft` ↔ 検証済み設定の変換
  - `toml_edit` 部分更新: コメント付き入力を与え、期待する出力文字列と突き合わせる
    （値の変更・追加・削除・並べ替えの 4 パターン）
  - キー記法の逐次検証と人間語表示
- 手動確認は受け入れチェックリストに Phase B 項目として追加する。最低限、次を含める
  - コメント・並び順の保持（往復編集）
  - 保存 → 即反映（リマップの挙動が変わる）
  - 壊れた入力が保存されないこと
  - GUI を開いたままリマップが遅延しないこと（不変条件 1・2。A-12 と同じ位置づけ）
  - 日本語入力（B-1〜B-8 の再確認）
